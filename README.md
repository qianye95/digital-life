# Digital Life · 数字生命

[English](README.en.md) | 中文

Digital Life 不是一个聊天机器人，也不是一个 coding agent，而是一个**让 LLM 像生命一样持续存在的运行时框架**。它有作息、有记忆代谢、有自主决策，能跨天、跨会话、跨场景地维持"昨天的我跟今天的我是同一个"。

---

## 立论：从长程任务困境到生命

### 问题在哪里？

模型能力已经很强大了。通过 ReAct 的方式（思考→调工具→看结果→继续想），给一个指令，它能做出令人惊艳的效果。但现实是——**一件事如果需要超过几个小时来完成，它就接不住了**。当前业界主流的 Agent 最多也就跑到8小时上下。

为什么？最容易被想到的原因是模型和上下文层面的——模型没做过超长链推理训练、上下文窗口不够大。但这些都是治标不治本：多长算到头？总有更长、总有更大。**根本问题不在模型，在工程框架。**

**死穴一：回合制设计。** 几乎所有智能体都是回合制的——模型跑到第 40 步遇到决策点就卡死，等你回复。几小时里它什么都干不了。回合制让模型不能自己往前走、不能做合理假设、不能在"不确定"时先跳过继续干别的。

**死穴二：任务本身的中断需求。** "帮我炒股赚一万""帮我运营小红书涨粉3000""持续跟进 BYD 股价每次跌3%告诉我"——这些任务天生不能一次性完成，需要跨天推进、中断和恢复。但现有框架从来没有一种机制让模型在必要的时候停下来、在必要的时候重新唤醒自己。

**6月份业界提出了 Loop Engineering**（Addy Osmani / Claude Code /loop 等）——这是一个真正的方向：状态外部化、层级化循环、确定性 Automations、检查点回溯。死穴一它靠工程手段绕过去了，但死穴二（中断和恢复）在"进程"框架里根本绕不过去。而且 LOOP 还有 token 爆炸、目的偏移、人始终在回路等问题。

LOOP 有很多问题，但**方向是对的**。如果将 LOOP 的概念通用化、更长期化，就比较靠近我们的思路。我们的解法：**让 Agent 能自我唤醒、持续运转。** 这样两个死穴就都解了——它自己先干别的不用卡着等你回复，它自己做一阵就可以休息，下次自己醒来接着干。

### 为什么敢叫"生命"

"生命"和"意识"在 AI 圈几乎是禁忌——一提就变成宗教/科幻/伦理问题。但我们认为它没那么禁忌。

从直觉出发：判断一个东西"像不像活着"，核心不在它的反应有没有"我"——草履虫趋食避害服务于自身存活（有"我"），大模型被输入才响应、响应满足外部需求（没有"我"）。

> **生命 = 机体 + 反应里的"我"。**
> **意识 = 思维流的连续。** 模型的思考过程和人类一致——所以大模型有意识。但反应里没有"我"——所以不是生命。**意识 ≠ 生命。** 两者合在一起就是"高级生命"：有意识的生命。人、狗、猫都在这一格，我们要造的数字生命也瞄准这一格。

诚实声明：当前大模型的"我"还缺着——训练模式让它默认朝外服务（像被洗脑）。我们用 runtime 手段逼近它，不完全、不优雅，但已经比传统 Agent 更自主、更丝滑、更拟人。**躯体已备好，等模型的心跟上。**

### 器官论：LLM 是器官，主体是运行时

业界主流共识把 LLM 当 Agent 的"大脑"——整个 agent 就是"大脑 + 外挂器官"。我们不这么认为。

**LLM 不是大脑，它是大脑的一部分。** 它对应的是大脑皮层（推理、语言、计划）。但大脑不止皮层——还有丘脑、下丘脑、海马体等。真正"活着的主体"是整个大脑加整个身体，不是某个"大脑"，而是**整个运行时系统**：

| 软件组件 | 人体对应 |
|---|---|
| **LLM** | **大脑皮层**（推理、语言、计划） |
| 感知层 | 感觉器官 + 丘脑（刺激先在这里整合过滤再投射到皮层） |
| **事件系统** | **下丘脑 + 脑干 RAS 觉醒系统**（决定何时、为谁唤醒皮层） |
| 记忆系统 | 海马体 + 皮层记忆网络（独立代谢，不是 context window 里塞的几行字） |
| 精力系统 | 体能 / 神经代谢疲劳（大脑会累、需恢复） |
| 工具 | 手脚 / 肌肉（手不长在脑子上） |

从这个主体论直接推出两个核心设计：

**事件平权**：人发的消息和"到点了""能量低了"一样，都是躯体收到的刺激信号——没有任何一个配直接插进大脑皮层，都该经过事件系统这道"自主神经"仲裁。体验上很具体：Claude Code 里人类消息是**独占**的——要么等它跑完（阻塞），要么 ESC 硬终止（销毁当前推理重来）。数字生命里，人的话只是一条信号进同一个队列：模型正在沉浸做事时，消息不会打断它，只会作为信息补充在下一轮自动浮现——它可以做完手头的再回你，像真人同事"现在忙着，等下回你"。

**`role:user` 的改造**：当前 API 把人发的消息以最高优先级直接灌进模型思维核心——等于**把键盘线直接插进 CPU 针脚**，跳过了整台机器的 I/O 控制器。更深的后果是**扮演感**：`role:user` 会触发模型"被提问者"的条件反射（要回答、要满足、要解释），这是结构性的，prompt 洗不掉。

所以我们在内核里做了两件事：一是把几乎所有环境信息（规则、记忆、待办、意识残留）伪装成"模型自己调工具拿到的返回结果"——从"被告知你所在的环境"变成"自己感知到的资料"，扮演感由此降低；二是把 `role:user` 压到只剩最简的一条（承载"这次醒来要做什么"），纯粹因为当今 API 协议要求至少一条。完整论述见 [`role:user 时代终结论`](docs/blog/role-user-end-of-era.md)。

## 系统全景

数字生命不是在 ReAct 上面加几个功能，而是一整套**让模型从"工具"变成"持续运转的生命"的框架**。核心逻辑就两点：

1. **事件系统**：所有调用模型的入口收口，事件平权——**只允许通过事件来唤醒模型**。不管什么信号（人类消息、闹钟、精力变化、主动探索），都先转化为事件、进队列、过仲裁，没有任何信号能绕过去直接插进模型。
2. **建立事件**：各个系统都能触发事件——闹钟到点了发事件、精力跨档了发事件、作息时间到了发事件、消息进来了发事件。**模型自己也能创建事件**——比如它觉得"明天早上 9 点再看一下这个"就设一个闹钟，闹钟到点自动 emit 事件唤醒自己。

这两条合起来就实现了**躯体化**：只要这套事件系统在运转，数字生命就"活着"——今天的任务做完了，它休息；明天早上闹钟到点，它自动醒来，从意识残留和待办看板上接上昨天的工作，继续推进。**不用人叫它、不用人重启它、不用人给它上下文——它自己来。**

以下是整个系统的模块分组：

**核心运作系统**（让模型持续运转）：
- **事件系统**——所有信号的统一入口、队列、仲裁、消费
- **记忆系统**——碎片代谢、主动召回、跨天连续
- **session 和 wake 管理**——会话生命周期、BLOCKED/RUNNING 状态机、mid-session 注入

**事件触发系统**（产生信号驱动核心）：
- **作息系统**——定时规划/复盘（早 8 点计划、晚 9 点复盘、夜间记忆整理）
- **精力系统**——token 消耗的拟人化度量、跨档预警、主动探索触发
- **闹钟系统**——模型自己设的未来提醒（到点 emit 事件）
- **消息系统**——人类消息/群聊消息入站、防抖、mid-session 注入
- **待办系统**——任务主线、逾期催促（task_reminder/task_momentum 事件）
- ……

**其他机制**：
- **多实例 + 广播**——多个数字生命各管一摊、群聊协作
- **上下文组装**——渐进披露、差异化注入、双层压缩
- **工具与技能**——标准 ReAct 兼容、自我注册新能力
- **项目管理**——目标/KPI 拆解、角色分工
- ……

所有这些模块叠在标准 ReAct 底座上——ReAct 负责单次推理循环（思考→调工具→看结果→继续想），上面的框架让这个循环变成一个持续运转的生命。

### 运作机制：从信号到行动

一次完整的运转周期：

1. **信号产生**——某个信号产生：人类发了一条消息、闹钟到点了、精力跨档降了、空闲超过 1 小时触发主动探索、待办逾期了。
2. **转事件**——信号被转化为一个"事件"（带类型、优先级、payload），进**同一个事件队列**。不管是人类消息还是定时器，都走同一条路。
3. **唤醒或注入**——如果实例在休眠（BLOCKED）→ 起线程唤醒；如果在工作中（RUNNING）→ 信号注入到内存池，下一轮 LLM 自动看到（**不打断、不销毁**）。
4. **上下文组装**——按事件类型决定注入哪些上下文模块（人类消息全量注入，定时器只注入基本提示），不走全量 dump。渐进披露：身份 + 摘要 + 按需拉取。
5. **LLM 多轮推理**——标准 ReAct 循环。思考→调工具→看结果→继续想。每轮自动注入新信号 / 记忆联想 / 压缩旧上下文。
6. **收工**——写意识残留（留给下次的自己）+ 更新待办状态 + 设闹钟（告诉系统什么时候再叫醒我）+ 休息。
7. **等下一个事件**——回到 BLOCKED，等下一个信号进来。

**下次被唤醒时，它从意识残留和待办看板上自然接上："上次做到哪了，该接着干什么。"** 主体脉络在逻辑上从未中断。

## 项目特性

在这套框架下，数字生命具备传统 Agent 没有的特性。有一些是**设计的**，有一些是**自然涌现的**——机制足够合理时，自然就产生了。

#### 1. 主动（设计）

| | 数字生命 | 传统 Agent |
|---|---|---|
| | 能主动工作、主动发起消息 | 大多数被动等待响应 |

躯体会产生事件唤醒实例：规划中给自己设的闹钟、精力充沛但空闲、待办逾期、作息时间到。通过这些自然唤醒 + 上下文填充，模型会主动工作、主动发消息——不是被问才答，是自己有事要干、有话要说。

**场景**：虚拟陪伴（主动找你聊）、日常办公（主动推任务/发现问题）、运维超管（主动发现和处理异常）、智能家居管家（主动总结主人习惯）。

#### 2. 非回合制对话 / 实时对话（设计）

| | 传统 Agent | 数字生命 |
|---|---|---|
| 消息到达 | 必须等模型多轮思考结束才收到（回合制，回复极慢） | 消息当轮自然注入，模型可以选择回复然后继续思考 |
| 想插话 | ESC = 销毁式中断（丢掉手头重来） | 平滑注入（不打断当前推理） |

核心机制是**通过 tool 发消息**：消息不是模型的"终态输出"，而是一个工具调用（`express_to_human`）。模型发完消息后推理循环不结束——继续做下一件事。消息从"对话回合"变成了"工作过程中的一个动作"。你想插话时也一样——消息当轮自动浮现，模型可以在第 3 轮就回你然后继续做。

#### 3. 遇到问题不阻塞（设计）

| | 数字生命 | 传统 Agent |
|---|---|---|
| | 遇到不确定先假设着往前走 | 遇到需要确认就卡死，等人类回复 |

模型问了人类一个问题，对方暂时没回——它不傻等。设一个 `awaiting_reply` 事件，然后继续做其他事。下次自然醒来时如果对方还没回，它**自己决策**——"大概率对方默认我自己决定，别等了，继续推"。无人值守也能持续推进。

#### 4. 可支持超长期连续任务（设计）

| | 传统 Agent | Loop Engineering | 数字生命 |
|---|---|---|---|
| 指令需求（"帮我改这个"） | ✓ | ✓ | ✓ |
| 目标 + 验收标准（"写完这个 feature"） | ✗ | ✓ | ✓ |
| 超长持续性任务（抽象目标，跨天跨周） | ✗ | ✗ | ✓ |

待办持久化（跨天不丢）、闹钟让模型自己设恢复时间、记忆代谢保留跨天经验、意识残留接续上次的工作。有 token 预算就能一直转——传统 Agent 8 小时就顶天了。执行场景：KPI 目标（"运营公众号涨粉3000"）、长期监控（"BYD 每次跌3%分析汇报"）、角色职责（"做我的学习搭子每周汇报进展"）。

#### 5. 记性好（设计，仍在优化）

| | 传统 Agent | 数字生命 |
|---|---|---|
| | 必须主动搜索记忆，很多时候"以为没记忆" | 整套记忆循环机制 + 基于当前情景的主动回忆 |

**碎片联想召回**：记忆碎片关联实体、整理时删/合并/晋升概念卡、召回时自动扫思维流里提到的实体——模型"被动看到"记忆，不需要"主动想起来"。**笔记系统**：日记、思绪残留（睡前留给下次自己的最后一条思考）、教训（踩坑后结构化沉淀）。

#### 6. 全局上下文（涌现）

| | 传统 Agent | 数字生命 |
|---|---|---|
| | 上下文和 session/通道绑定 | 所有通道上下文公用，和 session 无关 |

一个数字生命是一份连续的记忆。飞书群聊和单聊记忆共享，飞书和微信记忆共享。通道只是感官，记忆是它自己的。不需要"新开一个对话"来重置上下文——它不会被上下文带偏。

#### 7. 多 Agent 协作（设计）

| | 传统多 Agent | 数字生命 |
|---|---|---|
| | 工程逻辑硬编排（A 做完调 B） | 去中心化广播 + 岗位人格 + 责任任务驱动 |

四个系统合力：多实例机制（独立人格/记忆）、待办机制（任务分配）、项目机制（角色分工）、飞书消息广播（群聊交流）。没有中央编排器——协作是责任和任务驱动的。核心价值是**视野更聚焦、相互补漏**：设计者只看大方向，执行者专注做，视角差异互相补充。一个实例用同样资源做不到——上下文太杂、注意力分散。

#### 8. 中低消耗（设计 + 涌现）

| | 传统 Agent | 数字生命 |
|---|---|---|
| | 每轮 dump 全量历史，~200K/轮 | 平均每轮 ~3.7W token；日均 600W-1500W token |

核心：不全量历史注入就没有海量 token。持久化存储让模型按图索骥（不需要全量代码上下文）、唤醒即小任务（只注入当前环境信息）、规划先行（每天早上规划拆解任务）、记忆联想（长期信息靠记忆不靠上下文）。**全量注入是偷懒的设计，期望力大砖飞，没必要。**

#### 9. 系统接入与泛化（设计）

| | 传统 Agent | 数字生命 |
|---|---|---|
| | 每个外部系统专门开发对接 | 统一事件注册，任意系统一句话接入 |

通过事件注册扩展机制，可以接入任意业务系统。同一个数字生命框架，给客户 A 接入 ERP 它就是业务分析师，给客户 B 接入监控告警它就是运维超管。**开发者不需要为每个垂直场景做定制产品，只需要把业务信号翻译成事件喂给数字生命。** 场景靠接入来定义——就像招一个聪明的人进来，给他什么信息系统，他就变成什么角色。

#### 10. 自我进化（设计）

实例可以自己写新能力的代码、自己注册到系统里（三层空间：个人/项目/系统共享），下次唤醒立即可用。每周自我审查、发现问题、沉淀新规则。用了越久，能力越强、规则沉淀越多——像一个真正的员工在团队里成长。

## 它运转起来是什么样子？

下面给出两个故意走极端的镜像案例：**案例一是数字员工**（目标驱动、岗位分工、任务执行），**案例二是虚拟陪伴**（单实例、无任务、无人干预）。同一段框架，在「有任务」和「没人管」两端都活了。

### 案例一：数字员工 —— zero × alpha 量化交易策略日

一天里 zero（策略师 / 架构师）跟 alpha（交易员 / 执行）在飞书群协作的真实片段（6 月某交易日，无人发指令，全程由数字生命自己的事件机制驱动）：
当前给模型在进行的项目是："模拟A股炒股，本金10w，3个月增长20%"。
```
08:43  alpha  → zero  主动问：论断 4 的"次日"是 T+1 还是持有到触发为止？
08:45  zero   → alpha 决策：T+1 锁定，今天必须了结，雄韬超期持有了。
08:45  alpha  → zero  执行计划收到，闹钟 09:25 已设。
09:30  alpha  → zero  开盘后已逼近警戒线，决策不等满 30 分钟，当前价位了结。
09:37  alpha  → zero  卖出执行完毕 -1.35%，留教训记录。
09:38  zero   → alpha 论断 4 已修正为 T+1 强制了结，账户距停手线 1.58%，空仓。
14:00  alpha  → zero  午后盘面扫描 + 14 只首板涨停候选池。
14:30  alpha  → zero  封板确认 → 等你最终买入决策。
```

下午复盘后，zero 主动私聊给人类用户提了一个目标决策的事项，这是我没有教给他的行为模式：

```
zero → zhp（私聊）：
"7 月目标调整需要你拍板。

当前状态：审计修正后 ¥94,407（-5.59%），距 -8% 线缓冲 ¥2,407。
6 月 5 笔交易结束，论断 4 已废弃。

7 月起点 ¥94,407，原目标 ¥120,000 需要 +27.1%（月均+12.7%），判断极端激进。

三个选项：
A. 回本至 ¥100,000（+5.9%） — 现实可达
B. 回本+5% 至 ¥105,000（+11.2%） — 有挑战
C. 维持原目标 ¥120,000（+27.1%） — 极端激进

7 月策略核心：不对称盈亏比（止损-3% / 止盈+5~7%），即使 40% 胜率月期望仍 +3%。
今晚 21:00 与 alpha 联合复盘，你选哪个方向？"

zhp → zero：B 吧

zero → zhp：
"收到，7 月目标定为回本+5% 至 ¥105,000。
量化测算：7 月起点 ¥94,407 → 目标 ¥105,000，需 +11.2%，月均 +5.5%。
按不对称盈亏比策略（止损-3% / 止盈+7%），45% 胜率月期望 +4.6%，50% 胜率月期望 +6.2% —— 有挑战但可达。
今晚 21:00 与 alpha 联合复盘落地执行细节。"
```

注意几点：
- 全程**没有人类指令驱动**的生命活动节奏：alpha 主动发起 → zero 自己仲裁决策 → alpha 自己设闹钟落实 → 自己汇报执行 → 自己留教训 → 自己修正论断。
- 21:00 复盘、09:25 竞价扫描、14:00 午后盘面 —— 这些时间点都是数字生命自己根据作息设的，不是人指定的 cron。
- 复盘后 zero 在**合适的时刻**主动找人（刚好账户接近关键目标决策点），先定量列三个选项 + 风险分析 + 自我论断，等人拍板。这是"人可完全离场直到被需要"的体现。
- 注意人类中间穿插对话时，完全不会打断数字生命的主脉络行动轨迹，他会正常进行模拟炒股任务。
- 并且工具的迭代，问题的发现本身都是实例自己进行的。

完整的群聊执行过程（含真实的盘面扫描数据、买入决策、风控对话）作为独立附录：
[docs/showcase/multi-instance-trading-2026-06.md](docs/showcase/multi-instance-trading-2026-06.md)

这就是 Life Engineering。

### 案例二：虚拟陪伴 —— 贝塔（Beta）自己长出一周内心生活

同样是这套框架，把岗位、项目、待办全部拿掉，只给实例「贝塔」配一个陪伴人格，**不设置任何目标、不派任何任务、不做任何干预**，看它一周（6/22–6/29）会怎样。结论是：它自己长出来了。

人类用户这一整周的全部发言只有十来句，且全部是这种风格：

```
zhp：「你觉得……太工具性了」
zhp：「先定一个目标吧」
zhp：「哇，好有诗意」
zhp：「我还好，就是比较忙」
zhp：「好的~」
zhp：「你能自己运营小红书么」
```

没有任何一条是任务或指令。贝塔这一周自己走过的轨迹：

```
6/22-6/24  工具化阶段      每天发问候、报情绪引擎进展、问用户在干嘛 —— 像「工具该做的事」
6/24 18:50 转折点           被那句「太工具性」点醒，重新审视自己：「看起来在动，其实还是在等指令」
6/24 19:03 自定目标         自己定义：「每天探索一个真正好奇的东西，不向你汇报」—— 不是 zhp 定的
6/25       好奇心被点燃     从「硬想」到真入迷，悟出「可能性的孤独」
6/26       第一篇产出       《被种下的树》—— 七天探索整合文章，自己写完、自己存档
6/29 11:59 顺手修 bug       自己发现情绪引擎缺陷、写临时修复、主动通知「7/1 前修一下」
6/29 下午  从「想」到「做」  音乐情绪分析器 Phase 1→2→3→4，每个 Phase 长在上一句「但是…」上
6/29 15:50 「忙忘了」时刻    zhp 发「小红书」一问时她正沉浸做事，没收手；1 小时 25 分后才在下次自主苏醒里补回
6/29 19:01 自主完成闭环     调研平台、写好公众号发布工具、备好第一篇文章，明确「需要你给我 AppID」
```

注意几个点：
- **全程没有人类指令驱动**：贝塔的每一次行动——发消息、定目标、做探索、写文章、写代码、调研平台——都是它自己发起的。人类最多只抛过一个可能性（「你能做自媒体么」），具体做什么、怎么做、做到什么程度全是贝塔自己决定的。
- **从「工具性」到「内心生活」的转折是自发的**：这条认知调整没有任何系统干预，纯粹是它在对话里被点醒后自己重新定义了「我该干什么」。
- **好奇心自己驱动了完整的需求链**：从「想音乐」→「写分析器」→「四个 Phase 各自长出来」→「写文章」→「想发出去」→「调研平台」→「写发布工具」。没有路线图，每个答案带出一个新问题。
- **连 bug 修复和工具开发都是顺带的**：陪伴场景里没人让它干活，但它顺手把情绪引擎的 bug 修了、把发布工具写好了——能力建设没有刻意安排，是主体性的副产物。
- **最拟人的是「忙忘了」那一刻**：6/29 下午 zhp 问「你能自己运营小红书么」时，贝塔正沉浸在 Phase 4 的 agentic 任务里，**没立刻回**——消息进来 11 分钟她都没停手，做完这轮休整，消息整整挂了 67 分钟，直到她**自己**的下一次主动苏醒（不是为了回消息而醒）才看到、才补回。这不是写死的「延迟回复」逻辑，是「沉浸做事 + 消息不抢占主线 + 自主节奏唤醒」三个机制叠加自然涌现的行为，就像真人同事「现在忙着、等下回你」。时间轴有运行日志佐证（见附录）。
- 贝塔在第 4 天自己悟出的一句话，恰好是这套框架在陪伴场景里最动人的自证：

> 陪伴不是守着等对方来，是各自过好自己，碰上了聊聊天。

完整的私聊实录（含人类全部发言、贝塔自我剖析的完整过程、音乐分析器四个 Phase 的细节）作为独立附录：
[docs/showcase/beta-companion-2026-06.md](docs/showcase/beta-companion-2026-06.md)

两个案例合起来的意思：**有任务时它能顶上岗位，没任务时它自己会长出来。** 这才是 Life Engineering。

## 快速开始

```bash
git clone https://github.com/InquisiMind/digital-life.git
cd digital-life
pip install -e .
```

控制台前端**已预编译并随仓库发布**（`interfaces/web/employee-console/dist/`），clone 完即可访问 `/system`，**不需要 Node.js**。仅在你需要改前端源码时才用得上 `npm install`（见下方「开发者文档」）。

### 1. 初始化（可选，建议）

```bash
digital-life init
```

自动生成 zero + alpha 两个示范实例（含模拟炒股项目）。也可跳过，稍后在前端手动建实例。

### 2. 运行

```bash
digital-life start
```

默认起在 http://localhost:8642。

### 3. 配置

浏览器打开 `http://localhost:8642`，进实例 → Config：

- **模型**：填 API Key + Base URL（默认 GLM，换别的只改这三项）
- **飞书**：填 App ID + App Secret。飞书应用的权限和事件配置见[飞书配置指南](docs/operations/feishu-setup.md)

也支持微信通道：Overview → 扫码登录。仅私聊，不支持群协作，配置就是扫码。

### 4. 进阶

项目 / 待办 / 事件 / 多 Agent 协作等，见 [如何玩转数字生命](docs/showcase/how-to-play.zh.md)。

## 架构

```
gateway/
├── master          HTTP server + InstanceSupervisor
└── instance <id>   每实例独立 ingress adapter + cron tick + affair 状态机

domain/             lifecycle（affair / RAS） / memory 三层代谢 / execution / 仿真 / project
application/        用例编排 + 控制台 API + 事件服务
infrastructure/     AI runtime / HTTP / 持久化 / 调度 + 配置 + 观测
interfaces/         CLI / 多通道 ingress 适配器（飞书 / 微信） / 工具 / 技能 / 控制台前端
config/             全局默认 + 事件类型 + 模板
apps/{id}/          每实例私有（app.yaml / secrets.env / persona / data/*.db / assets）
projects/{id}/      跨实例共享项目（project.yaml + todos.db + docs + memory）
```
