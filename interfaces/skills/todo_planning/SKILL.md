# 待办拆解与规划方法论

## 什么时候需要这个 skill

当你面对一件事，出现以下任何一个信号时：

- **"这事我不知道从哪开始"**——说明它太大或太模糊，需要拆。
- **"这事好像要做很多步"**——脑子里能想到 3 个以上的动作，说明该结构化。
- **"这事我不知道做完了没"**——说明"完成"的标准不清晰，需要先定义产出。
- **"这事我一直拖着没做"**——可能不是懒，是潜意识知道它太大/太模糊，需要拆到能下手的大小。

如果你看到待办感觉"这个清楚、一两步能做完"——**不要用这个 skill，直接去做**。过度规划也是一种拖延。

## 核心方法：产出导向拆解

不是"把任务分成 A/B/C 三个部分"（那是按结构分，容易拆出伪步骤），而是**从最终产出倒推**：

### 第 1 步：定义"做完了长什么样"

不是"完成重构"，而是"代码能跑通测试、旧的登录逻辑被删掉"。
不是"做好研究"，而是"产出一份含 3 个方案的对比表"。
不是"搭建系统"，而是"有一个能跑的回测脚本，输入股票代码输出收益曲线"。

**定义产出时问自己**：如果有人来看我的结果，他"看到/拿到"的具体是什么东西？

### 第 2 步：从产出倒推必要的步骤

产出 ← 需要什么 ← 每一步又需要什么 ← 最先要做的是什么

```
目标：能跑的回测脚本
  ← 需要数据获取模块
    ← 需要确定数据源（这一步现在就能做）
  ← 需要策略逻辑
    ← 需要明确买入/卖出条件
  ← 需要回测框架
    ← 需要选一个框架或自己写
```

倒推出来的链，**最底层的就是你现在能下手的第一步**。

### 第 3 步：每一步要满足"可观测的完成"

拆出来的每个子待办，必须能回答："它做完了，我怎么知道？"

- ✅ "确定数据源为 akshare"（可观测：一句话结论）
- ❌ "研究数据源"（不可观测：研究到什么程度算完？）
- ✅ "回测脚本能对 1 支股票输出收益曲线"（可观测：能跑出结果）
- ❌ "搭建回测框架"（不可观测：框架到什么程度算搭完？）

**如果一个子待办没法定义"完成"，说明它还不够小，继续拆。**

### 第 4 步：安排顺序

大部分时候，倒推链已经给了顺序——从最底层往上看。但要注意：

- **有依赖**的按序做（先有数据才能回测）
- **无依赖**的可以挑精力最好的时候做
- **不确定的先做**——如果一个步骤的结果可能让你推翻整个计划（比如"数据源根本拿不到数据"），先做它，尽早失败

## 拆解后用什么工具落地

- 用 `todo_plan create` 给父待办添加有序步骤（适合有明确先后顺序的）
- 或用 `todo update(parent_id=父id)` 创建子待办（适合各自相对独立的）
- 两者区别：plan 是"一份计划清单"，子待办是"各自有完整生命周期的独立待办"

**简单选法**：如果是线性的（1→2→3），用 plan。如果每块自己就能独立跑、有各自的状态流转，用子待办。

## 反模式

- **"为了拆而拆"**——两步就能做完的事，硬拆成六个步骤。拆是为了能下手，不是为了看起来有计划。
- **"拆完不执行"**——创建了一堆 plan 步骤，但每次醒来都重新规划，从没真正做过第一步。
- **"拆出伪步骤"**——"研究""分析""优化"这类词没有可观测的完成标准。逼自己说具体。
- **"一次拆到底"**——第一层都没开始做，就拆到第三层第四层。先拆到能动手的粒度，做起来再说。
- **"拆完就锁死"**——做着发现计划不对还硬按原计划走。计划是活的，执行中发现该调整就调。
