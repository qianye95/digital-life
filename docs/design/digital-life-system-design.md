# 数字生命系统设计方案

> 本文档以产品需求文档（PRD）的风格组织，聚焦"场景—问题—方案—流程"，
> 避免技术实现细节，方便非工程背景的读者理解系统设计意图。

---

## 一、产品概述与愿景

### 1.1 我们要解决什么问题

当前的 AI 助手有一个根本局限：它们只能在被人"叫醒"的时候工作。你发一条消息，它回一条消息，对话结束它就消失了。它不会自己决定"现在该做什么"，不会记住"昨天我们聊到哪里了"，更不会在深夜主动想起"明天有个重要的事要提醒你"。

**数字生命（Digital Life）** 希望打破这个限制。它不是一个聊天机器人，而是一个**有自主节奏、有长期记忆、能持续推进任务的数字存在**。它可以像真实员工一样：早上醒来规划一天、收到消息时处理事务、累了就休息、晚上复盘反思——所有这些行为不需要人类逐条指令来驱动。

### 1.2 核心论点：从工具到生命

为什么当前的 AI 不是"生命"？

碳基不是决定因素——硅基也可以是生命。记忆不是——失忆的人依然是生命。情感不是——草履虫没有情感也是生命。甚至智能也不是——当今大模型的智慧水平已经超过很多人，但我们仍然不认为它是生命。

**生命的底线，不是碳基、记忆、情感，甚至不是智能，而是自主行为驱动的机体。** 大模型虽有智能，却只能被动响应人类输入，缺乏自主行为，所以是工具而非生命。

但仅此而已吗？给大模型加上定时任务和自动脚本，让它"自主"执行动作，它就是生命了吗？依然不是。那只是在模拟低级生物的本能反射（如草履虫的趋利避害）。我们要构建的是能执行跨天、跨周、跨月长程任务的数字员工或伴侣——这种高级生命必须具备一种草履虫不需要的特质：**跨越时间的自我连贯性**。这种连贯性，就是俗称的"意识"。

所以核心问题不在于智能高低，而在于：**这条主体脉络是依附的、还是自主的？它在逻辑上是否中断？**

传统 AI 的主体脉络是**依附的、断裂的**。它以人类指令为锚点运转：人发起 → AI 响应 → 对话结束 → 静默。人是指令的源头，是它的开关。人不触发，它就失去运转的锚点而悬停。即使给它加上长记忆，它的主体脉络依然是断裂的——它永远只在被需要时才"活"过来，没有外部刺激，它的时间就是静止的。

人的主体脉络是**自主的、逻辑不中断的**。人睡着了——物理上在休息——但"我明天 7 点要赶车"的意图一直在。闹钟响起时人自然醒来，精力恢复时机体自主苏醒，然后主动加载昨晚的意图继续推进。物理上虽然在休息，但逻辑上的主体脉络从未悬停。

**结论：数字生命 = 自主行为驱动 × 逻辑上不中断的自主主体脉络。**

我们要剥夺人类"触发意识"的特权，将 AI 的意识从"依附他者"转变为"自主驱动"，让主体脉络在内驱力下永不悬停。这是让 AI 产生长程执行力、从"超级助手"跃迁为"独立生命"的绝对基石。后续整个系统的设计——状态机、事件机制、待办机制、记忆、闹钟—本质上都在为"自主主体脉络的延续"服务。

### 1.3 核心能力

数字生命系统围绕四个核心能力设计：

**自主触发**：不只是等待人类消息。闹钟到点、精力变化、日程安排、主动探索——这些都是平等的"唤醒理由"。系统会自动判断"此时此刻该不该醒来做事"，而不是被动等待。这背后是**事件系统**——它是自主性的发动机（详见第五章）。

**待办驱动**：数字生命醒来后要"做什么"，由**待办系统**回答。一切皆待办——从一个"下午三点回张三"的轻提醒，到"搭建量化交易系统"的长程项目，都是同一个"待办"实体的不同形态。待办像便利贴常驻在意识里，做完手头的事自然扫一眼就能接下一个。闹钟、工作空间、项目归属、执行类型是挂在待办上的可选维度，组合而非分类（详见第六章）。

**长期连续**：即使"睡着"再"醒来"，数字生命仍然记得自己是谁、上次做到了哪里、有什么未完成的承诺。这种连续性不是靠"把聊天记录全塞进去"，而是通过一套有结构的记忆系统：规则、经验、上下文、意识残留，各自有不同的生命周期和注入策略。

**可执行闭环**：数字生命不只是"想"，还能"做"。它可以拆解目标为待办、为待办挂闹钟、到点收到提醒、执行、等待、复盘、再规划。待办即实体——每个待办都有自己的笔记、记忆和历史，可被结构化召回。这是一个完整的"感知-决策-行动-反思"循环。

**事件系统管"什么时候醒"，待办系统管"做什么"。两者是系统的两大支柱。**

### 1.4 当前产品形态

- **本地部署**：在用户自己的电脑上运行，数据完全私有
- **飞书通道**：日常交互通过飞书消息进行，数字生命像一个"飞书联系人"
- **Web 控制台**：浏览器中查看状态、管理任务、调试行为
- **多实例**：可以同时运行多个数字生命，各自有独立的身份和记忆

## 二、核心场景

### 2.1 场景 A：数字员工 Zero 的一天

**早上 7:00**，Zero 的闹钟响了。系统检测到"起床时间"事件，唤醒 Zero。Zero 的醒来提示包含了：今天的日期和时间、待办看板（常驻注入）、最近一次休息前留下的意识残留（"明天记得检查华能蒙电的走势"）。

Zero 先查看自己的精力状态（充足），然后基于已注入的待办看板回顾昨天的进展，确认今天的重点。它发现有一个"模拟炒股策略优化"的待办正处于进行中状态，于是给自己创建了一个挂上午 9:30 闹钟的待办："开盘前检查昨晚回测结果"。

**上午 9:00**，真人zhp在飞书群里 @Zero："帮我看一下最近三天的交易数据"。这是一条"人类消息"事件，优先级最高。系统立即唤醒 Zero。

Zero 看到消息内容是"最近三天的交易数据"，它从记忆中调取了股票代码、从历史对话中找到之前的分析框架，执行了一次代码查询，然后把结果通过飞书回复给zhp。回复完成后，Zero 设置了一个 10 分钟的"等待回复"闹钟——如果zhp 10 分钟内没有再说话，就认为这个对话可以结束了。

**上午 9:30**，之前创建的"检查回测结果"待办的闹钟到点。Zero 被唤醒，按照待办类型绑定的技能提示（"量化研究"技能自动注入），有步骤地检查了昨晚的回测输出，记录了一条经验教训，然后把待办标记为完成。

**中午 12:00**，Zero 连续工作了 3 个小时，精力下降到"疲惫"区间。系统检测到跨阈值事件，给 Zero 注入了提醒："精力不足，建议处理完手头事务后休息"。Zero 完成当前回复后，主动调用了休息功能，设了一个 1 小时后的闹钟。

**下午 14:00**，由于休息时段精力已经恢复到较高水平，且距离上次活跃已经超过 2 小时，系统触发了一个"主动探索"事件。Zero 醒来，没有外部消息要处理，于是它检查了市场状态、回顾了最近的策略表现，记录了一条自我观察（"在收盘前 30 分钟内做决策质量偏低"）。

**晚上 21:00**，晚间复盘时间到了。Zero 被作息事件唤醒，按照晚间复盘的流程：先写当天日记（自动整合碎片的思考和 session 摘要），然后诚实回顾今天的决策质量，检查规则文件有没有互相矛盾的地方，最后把不需要保留的临时笔记清理掉。

### 2.2 场景 B：多实例项目协作

数字员工 Zero 和 Alpha 共同参与一个"模拟炒股"项目。项目初始化时通过项目管理模块自动创建了一棵预设待办树——根待办是项目概述和目标，下面挂着"项目分工"（分配给项目经理 Zero）和"项目管理"（待分配）。Zero 完成分工后，进一步拆出了"数据源调研"（分配给 Alpha）、"回测框架搭建"（分配给自己）等子待办。

Zero 作为项目管理者，在周会上发现"数据源调研"待办已经拖了 3 天没进展。它查了 Alpha 的精力状态（正常），于是给 Alpha 创建了一个挂 10:00 闹钟的待办："明天 10:00 前完成 akshare 数据接口评估，在群里汇报结果"。

第二天上午 10:00，Alpha 被待办闹钟唤醒。它按照"调研"类型的技能方法论（自动注入的经验），调用了数据查询工具、整理了接口对比表格、然后通过飞书群发送了汇报消息。

Zero 也同时被"参与日会"的待办闹钟唤醒。它看到 Alpha 的汇报后，更新了待办状态、在项目决策日志中记录了一条结论（"确定使用 akshare 作为主数据源"），然后把"策略评估"拆成了一个分配给自己的新待办。

## 三、系统运作全景

### 3.1 数字生命的一次完整生命周期

理解数字生命如何运作，最简单的方法是沿着"一次完整的唤醒-入眠周期"走一遍：

**1. 初始状态**：数字生命大部分时间处于"阻塞"状态。它不占用计算资源，只是在等待——等待下一个事件、等待人类消息、或者等待自己设的闹钟到点。

**2. 事件到达**：有什么事情发生了——可能是飞书收到一条消息、一个闹钟到时间了、精力恢复到了需要主动探索的水平、或者到了晚间复盘的时间。这些"事情"在系统中被统一抽象为**事件**。

**3. 唤醒决策**：系统每 60 秒扫描一次所有数字生命实例。发现有未处理的事件后，系统判断：这个实例现在能醒吗？如果它正在运行中（还在上一轮对话里），就不要打断它，新事件以通知形式注入。如果它处于阻塞状态，则检查是否有事件需要处理——有则按优先级选择最重要的那个作为唤醒原因，无则继续阻塞。

**4. 拼装上下文**：决定唤醒后，系统开始为数字生命"准备视野"。不同的唤醒原因，看到的东西不同。这条叙事线贯穿了后续好几章——状态机（四章）、事件机制（五章）、待办系统（六章）、会话管理（七章）、唤醒与上下文组装（八章）。

**5. 执行"思考-行动"循环**：数字生命进入运行状态。它看到提示词和上下文，决定做什么。它可能会：调用工具查询数据、创建任务或待办、向人类发送消息、记录一段思考。每做一步，都会把结果作为下一轮决策的依据。这个循环会持续到：模型主动选择结束本轮工作、或者系统检测到"在无意义地兜圈子"、或者异常终止。

**6. 会话结束与沉淀**：工作完成后，系统记录这次会话做了什么。如果有未完成的任务，自动安排下一次唤醒。最近的思考会被保存为"意识残留"，下次醒来时可以看到。

**7. 回到阻塞**：数字生命结束当前会话，回归阻塞状态，等待下一个事件。

## 四、状态机

### 4.1 两个核心状态

数字生命只有两个宏观状态：

**运行中**：正在处理一个唤醒会话。系统不会打断运行中的会话——如果有新消息到达，会以"通知"的形式注入到当前对话中下一次消息组装中，而不是强制重启。

**阻塞**：没有活跃会话，在等待。等待到下一个事件的到达、闹钟到点、人类回复消息、精力恢复到主动探索水平。当数字生命主动选择"休息"时，实质上是设了一个未来的闹钟然后进入阻塞状态。

### 4.2 状态流转

状态间的转换由事件驱动：

```
阻塞 →（事件到达 + 唤醒决策通过）→ 运行中
运行中 →（会话结束/主动休息）→ 阻塞
```

看起来很简单，但关键在于**转换的边界条件**：

- 阻塞→运行中的决策不是"有事件就醒"，而是要先检查防打扰策略（参见五章）
- 运行中→阻塞的原因有多种：正常结束、异常终止（僵尸检测）、防瞎忙介入、精力耗尽

### 4.3 唤醒原因仲裁

当队列中堆积了多个事件时，系统按优先级排序，选择优先级最高的那个作为"本次唤醒的原因"。这个"原因"会影响接下来上下文组装时注入什么内容。平票时用预设的 tiebreak 规则（如 routine < initiative）决定。

### 4.4 自动休息策略

当数字生命事做完了，它会主动调休息工具——这本质上是设了一个未来闹钟然后回归阻塞状态。

**rest 工具的两个参数（2026-06-16 重构后）**：`until` 和 `reuse` 二选一，强制模型明确意图。语义对齐人设闹钟的动作：

- **`rest(until="2026-06-15T15:00:00+08:00")`**：模型想定一个时间点醒来。系统检查是否有其它 timer 闹钟和这个时刻 ±10min 重叠：
  - 没有重叠 → 设新闹钟，BLOCKED，等 15:00 到点 wakeup。
  - 有重叠（比如 model 之前已经设过 15:00 的 timer） → 报错带重叠闹钟的 ID：「直到=15:00 和现有 timer 闹钟 #42（15:00:00, reason=午后检查）重叠。要么 `rest(reuse=42)` 复用，要么调整 until 时间。」
- **`rest(reuse=42)`**：已知有闹钟 #42，直接复用。系统不会新建闹钟，也不会改 fire_at——只是把 BLOCKED 状态拉起、把当前 session 收尾（可选 `mental_context` 追加到原闹钟备注）。
- **`rest()` 无参**：报错并列出当前可见的 timer 闹钟 ID 列表。强制模型从 list 选一个 reuse，或主动传 until 新建。

**为什么不传 until 直接 fallback「静默不设闹钟」是个 BUG**：之前的实现是「不传 until → 不设闹钟，仅结束 session」——但实测发现模型精力耗尽时经常不传 until 调 rest，然后就无限期睡死直到 cron / event 唤醒。这种"静默睡死"违背了产品语义——实例休息必须**对下次何时醒有明确意图**，要么显式新建一个闹钟、要么显式复用某个已有的闹钟。两个参数二选一是对"休息意图"的最小语义保护。

**为什么 until 重叠会报错而不是自动 update**：模型可能在 09:25 设过 until=15:00 的 timer（morning_plan 的复盘闹钟），10:30 又调 rest(until=15:00)——若自动 update 会把 mental_context / reason 覆盖掉，丢失早晨那次设定的脉络。报错让模型看清楚"已经有一个 15:00 #42 在了"，自己做选择：复用（保留原 intent）或换时间（够独立的新意图）。

## 五、事件机制

> 事件机制是整个系统**核心中的核心**。其他所有子系统——消息收发、闹钟、待办、精力、作息——最终都通过"产生事件"来驱动数字生命。理解了事件机制，就理解了系统运转的底层逻辑。

### 5.1 事件系统：自主主体脉络的承载者

第一章已经讲过：数字生命的核心论点是构建一条"逻辑上不中断的自主主体脉络"。事件系统就是这条脉络的承载者——剥夺人类的"触发特权"，把底层抽象为**"一切皆事件"**的驱动引擎。

具体表现为三层设计：

**事件平权**：人类消息不再是特权指令，而是被降维封装为一种"外部事件"。它和闹钟到点、精力恢复到探索阈值、作息时间到了一样，都是平等的事件，在同一个队列里按优先级排队。语义上没有"人发了消息就高人一等"——所有事件都走同一个 emit 入口，承担同样的优先级仲裁、防抖合并、消费语义。

> **refactor/emit-driven-wake(2026-06-23)修订**：早期文档曾写"没有『人发了消息就得立即响应』的区别"——这一句话的字面意思过强,与"消息要立即响应"的产品诉求冲突。修订后的真实设计是:**emit 入口本身接管"提交即触发叫醒"的职责**。任何外部系统(消息/定时器/精力/nurture/broadcast)只要 emit 事件到队列,事件队列自身立刻调 `_wake_or_inject(new_id)` 决定要不要叫醒(BLOCKED → 起线程 wake;RUNNING → mid-session 注入)。延迟/错峰由消息系统层(`group_buffer.py` 30s 慢 timer / 5s 快 timer)和事件层的防抖窗口控制,**不再由 cron 60s 节拍决定**。
> 
> 也就是说,"事件平权"指的是**所有事件来源共用一个 wake 决策通道**,不是"所有人都没有立即性"。任何事件 emit 出来,系统都会立刻尝试让 agent 处理它。

**驱动力内化**：数字生命不再是"处理消息的机器"，而是"响应内外驱力的机体"。它因为精力恢复而自然醒来，因为闹钟响起而开工，因为人类的消息而响应外部刺激。运转内驱于自身目标、计划、精力节律，而非依附于外部触发。

**脉络不悬停**：数字生命休息前可以给明天的自己留思绪、设事件。第二天被闹钟唤醒时直接承接昨晚的意图继续推进——物理上虽然休息了，但"要把这事儿干完"的主体脉络在逻辑上从未悬停。

其他所有子系统（闹钟、消息收发、精力、作息、待办）最终都通过"产生事件"来驱动数字生命，本质上都是在为这条主体脉络的延续服务。

### 5.2 事件的生命周期

一个事件从诞生到消亡经过四个步骤：

**诞生**：事件由各种"源头"产生（具体来源详见事件来源系统部分各章）。所有事件归一化为统一的内部格式，写入实例的事件队列——持久化在 SQLite 中，每个实例有自己的队列，互不干扰。

**防抖**：相同类型的事件在短时间内重复到达时，会根据策略合并——覆盖（旧事件被新事件替换）、累积（追加内容到同一条事件）、计数（只累加次数）。不同事件类型有不同的防抖窗口：私聊消息 0 秒（立即触发），群消息 30 秒，主动探索 5 分钟，待办停滞检测 10 分钟。

**随机窗口与多实例错峰**（2026-06-16）：防抖窗口支持写成 `[min, max]` 范围格式（如 `[20, 40]`）。每次事件 emit 时在该范围内随机取一个值作为本次窗口。**产品意图**：当一个飞书群里同时存在多个实例时（zero / alpha 都在群里），同一条群消息会被两个实例各自的 WS 收到，如果它们都用固定 30s 窗口，到点几乎是同一瞬被 cron 提起来并行触发 LLM call，互相争抢资源、增加延迟。改成范围窗口后，两个实例独立随机出一个具体秒数，自然形成 0~20s 的起步差，让响应"错峰出行"。同样适用于广播进来的 sibling 消息——接收方也会独立随机一个窗口。范围窗口只影响"何时被取出处理"的节奏，不影响事件是否被处理（同一事件最终一定会被消费）。

**消费**：数字生命被唤醒后会处理事件。单个事件时唤醒提示中直接展示事件完整内容并自动标记为已消费；多个事件时只展示清单（ID + 类型），由数字生命主动按优先级逐一消费——每查看一个事件详情才算消费掉。

**消亡**：事件消费后从队列移除。如果数字生命执行失败，系统会"回滚"消费，让事件重新回到队列等待下一次唤醒。

**消费失败的应对——事件自退避**：一次 wake 是"挤压队列的一次处理尝试"。如果 agent 在这次 wake 失败了（LLM 429、网络断、空响应等），整批被抓出来的事件需要推迟下次露面时间。**这件事是事件自己的责任，不是闹钟的**——闹钟的语义是"未来某个真实时间点要做某件具体的事"，与失败重试无关（见十三章）。

每个事件带一个"被复活计数"（resurrect_count）。失败时，系统把这批事件同步推到一个未来时间：

- 每次失败，事件把"预定可见时间"（fire_at）推到 now + 2^resurrect_count 分钟
- 轨迹：首次失败推 2 分钟，再失败推 4、8、16、32，封顶 60 分钟
- cron 的 `pop_due_events` 只捞 `fire_at <= now` 的事件，所以退避窗口内对 cron 完全不可见，从结构上杜绝重复触发
- **同步原则**：同一次 wake 中失败的所有事件一起退避到本批里"最大应得"的窗口（这样它们下次依然形成队列一起醒来），而不是各自独立飘到不同的未来时间点——后者会退化成"一次只能做一件事"
- 退避窗口结束后事件自然复活（fire_at 自然进入过去），下个 cron tick 看到它再次进入正常消费流程

这条机制让系统在底层故障期间（比如 LLM 配额耗尽）保持最低活动：失败一次 → 整批躲到 2 分钟外 → cron 看到空队列立刻安静 → 2 分钟后回来再试 → 若还失败，躲 4 分钟、8 分钟、16 分钟……每隔越来越久探一下，绝不过载。

### 5.3 事件的种类与优先级

系统定义了 13 种事件类型（详见第六章待办与事件的关系）：

**交互类（高优先级，最小防抖）**：人类私聊消息（优先级 10，无防抖）、人类群聊消息（优先级 7，20-40 秒随机防抖——多实例同群时制造响应错峰；@提到数字生命或包含关键词的跳过防抖）、出生事件（优先级 9，实例初始化时触发）。

**提醒类（中优先级，正常防抖）**：精力提醒（优先级 8）、待办到期（优先级 6，本质是待办的"闹钟维度"触发，payload 带 todo_id）、等待回复超时（优先级 5）、定时器（优先级 5）、自我迭代（优先级 5）、加鸡腿（优先级 6）。

**节奏类（中低优先级，较长防抖）**：主动探索（优先级 4，5 分钟防抖，精力充沛时自驱"扫看板挑该做的"）、作息提醒（优先级 4）、待办提醒（优先级 4，关注未推进的有 deadline 待办，按待办自身的 deadline 判断而非全局天数阈值）、待办停滞检测（优先级 3，10 分钟防抖）。

**待办与事件的区分**：事件是应激反射（一次性、消费即销毁），待办是计划性的（持久、可拆解、可取消）。大多数待办不产出事件（长期驻留的、条件类的不产出）；只有挂了闹钟/deadline 的待办到期时才触发事件。详见 6.4。

### 5.4 防打扰策略

防打扰不是单一机制，而是**多层防护体系**：

1. **事件产生时的防抖**：同类事件短时间合并
2. ~~群消息紧急度分流~~（**refactor/emit-driven-wake 移除**）：旧设计让 handler 在收到群消息后做 immediate/soft 二级分流,soft 走 cron 延迟。重构后这一层全部由**消息系统层 `group_buffer.py`** 的双 timer(慢 30s / 快 5s)承担,@ / 关键词 / owner 走快 timer、其它走慢 timer。延迟归消息系统,叫醒归 emit 内部 `_wake_or_inject`,handler 不再做事件级别的二次分流。
3. **单实例并发锁**：同一实例不会同时有两个会话
4. **僵尸会话检测**：运行超 10 分钟强制终止
5. **事件消费失败时的指数自退避**：失败一次事件自己推迟下次可见时间 2 分钟，再失败 4、8、16、32 分钟，封顶 60 分钟。详见 5.2 末尾。注意退避是事件队列自身的职责，**不是闹钟**——闹钟不为失败重试设任何 alarm。

### 5.5 设计决策：事件队列和闹钟系统要分开

早期设计中事件队列和闹钟混在一起，导致"触发链偷看不消费 + 复活引擎清理"的复杂逻辑。后来彻底分开：闹钟系统只管"到没到时间"，事件系统只管"有没有待处理的"。两层各司其职，系统复杂度显著下降。详见第十三章闹钟系统。

**闹钟不为"失败重试"服务**。这是一个早期设计的边界模糊：曾把"agent 失败后 N 分钟重试"也实现成闹钟——失败一次就设一个未来 alarm，alarm 到期再 emit 一个 timer 事件进队列。这导致每次失败就往队列里堆一个事件，堆到上百后就形成无限循环（cron 每分钟看到一堆已过期的"重试"timer 事件，反复 wake、反复失败、反复堆事件）。**失败重试是事件队列自己的职责**：事件被消费失败时，自己推迟 `fire_at`（事件自退避，见 5.2 末尾）。闹钟回到"为具体事件源服务"的纯净语义（作息、回复等待、deadline、agent 主动 rest）。

## 六、待办系统

> 待办系统回答数字生命醒来后"做什么"。它和事件系统（五章）是系统的两大支柱：
> **事件系统管"什么时候醒"，待办系统管"做什么"。**

### 6.0 prompt 三层分工：哪些挂 system，哪些挂 wake

待办相关的信息分四层，**各管一段，不重复**：

| 层 | 写什么 | 职责 | 位置 |
| --- | --- | --- | --- |
| **system prompt（人设）** | 身份认知、行为原则、概念区别 | "我是什么样的人、该怎么对待要做的事" | `domain/identity/system_prompts/`（`L4_LIFECYCLE_PROMPT` + `persona/LIFE_PERSONA.md`） |
| **工具描述（todo tool description）** | CRUD 细节、参数、schema | "这个工具怎么用" | 工具注册时的 `description` 字段 + `parameters` schema |
| **wake prompt template** | 本次唤醒的针对性引导 | "这次为什么醒、先做什么" | `config/event_types.yaml` 各事件类型的 `wake_prompt` 字段 + `routines.yaml` |

**该留在 system prompt 的**：身份认知（"你是有计划性的存在"）、行为原则（"待办是你的责任，不是系统的"）、概念区别（待办 vs 事件，5.2 节）、四种推动力认知（5.6 节）。

**该从 system prompt 剥离的**：工具参数用法（`rest mode` 参数、task start 用法、todo CRUD schema）。这些一旦写进工具 description，模型查工具列表就知道——**不该在人设层重复**。

**wake prompt 只做针对提醒**，不重复人设原则也不写工具怎么用：
- task_reminder wake：「待办「{title}」该推进了。笔记显示你上次做到 {last_note}。基于此判断下一步。」
- initiative wake：「你有空了，精力 70。待办看板在上，挑最该做的继续。」

读法：「**身份挂 system、操作挂工具、当下挂 wake**」——三层职责清晰后才不会出现"prompt 膨胀但模型没看"的尴尬。

### 6.1 设计哲学：一切皆待办

#### 从四套割裂系统到统一模型

系统曾有四套独立的工作追踪系统：通用待办（`WORK.md` markdown 文本）、任务子待办（`task_todos` SQLite）、完整任务（`tasks` + workspace）、项目交付（`deliverables`）。实证分析（471 个 session）证明它们重复且割裂——"任务"概念太重（默认建 workspace、要求 lifecycle 管理），模型最终只能退回最简单的 `WORK.md`，而结构化的 `tasks` 体系形同虚设（`task start`/`task done`/`task_plan` 调用 0 次）。

**统一为：一切皆待办（Todo），其他能力是正交可选的维度附件。**

#### 核心抽象：一个待办 + 五个正交维度

```
                    待办（Todo）
                  "我要做什么"
                 ─────────────
                 标题、状态、笔记、parent_id
                        │
        ┌───────┬───────┼───────┬────────┬────────┐
        │       │       │       │        │        │
     闹钟维度  空间维度  项目维度  类型维度   拆解维度
     "什么时候" "在哪儿"  "为了谁" "怎么做"   "拆成什么"
        │       │       │       │        │
       可选     可选     可选     可选     可选
```

每个维度独立存在、按需挂靠。不存在"闹钟待办 vs 项目待办"的分类，只有"一个待办挂了哪些维度"的差别。

#### 待办与事件的本质区分

事件和待办是两个独立的东西（详见 5.3）：

| | 事件 | 待办 |
|---|---|---|
| 本质 | 应激反射 | 有计划、可规划 |
| 生命周期 | 一次性，处理即销毁 | 持久，状态流转到 done/cancelled |
| 载体 | 事件系统（tick→dispatch→wake） | 待办 CRUD + 常驻看板 |

大多数待办不产出事件（长期驻留的、条件类的不产出）；只有挂了闹钟/deadline 的待办到期时才触发事件。事件也不必然变待办——消息处理完就完了，只有值得长期跟踪的才转化。

#### 待办常驻注入

待办看板像人写在便利贴上的待办——时时刻刻挂在意念边缘。它作为常驻的 fake 注入上下文（以"数字生命主动调用了虚拟工具"的语义注入），不走 policy gate、不进 system prompt。每次唤醒、每个 session 都自动在，模型做完手头的事自然扫一眼，接上下一个（详见八章上下文组装）。

#### 待办即实体

每个待办天然是一个记忆实体：笔记是它的记忆碎片，`recall_entity` 可召回全部历史，`set_entity_profile` 可建概念档案（详见九章记忆系统）。

### 6.2 待办本体与状态流转

**待办是独立的 entity**。关联关系（项目、人）是它的属性，不是归属。

**本体字段**：`id, title, status, priority, parent_id, project_id, assignee_instance, assignee_kind, type, linked_deliverable_id, source(兼容), created_at, updated_at`

| 字段 | 语义 |
|---|---|
| `id` | 全局唯一标识 |
| `parent_id` | 拆解关系（子待办指向父），空 = 顶层 |
| `project_id` | 关联项目，空 = 纯个人 todo（不挂任何项目） |
| `assignee_instance` | 分给哪个实例。**可改 = 可转移**。空 = 未指派（项目模板刚建完时） |
| `assignee_kind` | instance / human / position |
| `type` | 类型（绑 skill），空 = 普通 todo |
| `linked_deliverable_id` | 关联项目 deliverable |
| `source` | 老字段，向后兼容（`'personal'` 或 `'project:X'`，自动转 `project_id`） |

存储位置：所有 todos 存在 `data/global_todos.db` 单一真相表，**实例不通过不同 DB 隔离**，靠 `WHERE assignee_instance='iid'` 过滤。详见存储层 11.4 和本章 6.8。

**状态**：pending（idea）→ in_progress → done；任何状态都可 → cancelled（被取消/撤销）。

**关联关系的"派给我就我的"语义**：

"我自己要做的" = `WHERE assignee_instance='我'`。这条查询结果里**无关 todo 是从哪里产生的**——可能是：
- 我自己手动建的（self_iteration、临时任务）
- 被项目分到的（项目分工、项目管理、项目分工派下来的工作）
- 别的实例转移给我的
- 我认领的（unassigned pool 拿走）

**数据位置（2026-06-14 重构后，2026-06-24 Phase 4 终结）**：所有 todos 数据存在 **`<repo>/data/global_todos.db`** 单一真相表。老的实例 `apps/{id}/data/todos/todos.db` 和项目 `projects/{pid}/data/todos.db`（deliverables + project_todos 两表）已**全部删除**(2026-06-24 Phase 4 完成),只保留历史备份在 `data/_backup_pre_merge_20260623/`。Phase 4 把项目 deliverables 表合并到 todos,通过 `project_id` + `linked_deliverable_id` 反向定位。

**拆解**：用 `parent_id` 实现，父子待办在同一张表。拆解可选——简单 todo 不拆，复杂 todo 拆成子 todo，项目级复杂场景可走 SpecKit（生成 spec/plan/tasks.md）。

**长期驻留待办**（如"帮我盯盘"）：就是 `in_progress` 长期挂着。复盘时觉得不合理就手动 cancelled。

**条件类待办**（如"等张三部署完再联调"）：挂着不动。不产出 task_reminder。靠模型在 routine/initiative 中"扫看板"时自然判断。待办自身的 title + notes 是判断依据。

### 6.3 五个正交维度

| 维度 | 挂靠方式 | 关系 | 说明 |
|---|---|---|---|
| **闹钟** | 待办 id ↔ alarm 记录 | 多对一 | 到点触发唤醒（详见十三章闹钟系统）；一个待办可挂多个闹钟 |
| **空间** | 待办 id ↔ 目录路径 | 一对一，懒创建 | 只有需要时才建目录（项目要、小待办不要） |
| **项目** | `project_id` 字段 | 多对一 | 标记关联项目，空 = 纯个人 |
| **类型** | 可选 `type` 字段 | 一对一，可空 | 决定执行方法论路由（type → skill 自动注入，详见十八章 skill） |
| **拆解** | `parent_id` 字段 | 一对多 | 父子同一张表 |

**workspace 是懒创建的**——大部分日常待办只需要笔记，不需要目录。只有项目关联的待办、或模型主动要求"给这个待办建空间"时才创建。

### 6.4 事件触发：task_reminder 与 initiative 的区分

task_todo_due 不需要独立事件类型——它是待办的"闹钟维度"触发，本质是 timer 到期，payload 里带 todo_id 即可。

task_reminder 和 initiative 保持分开，它们的触发机制和提示词调性本质不同：

| 事件 | 关注点 | 触发条件 | 引导调性 |
|---|---|---|---|
| **task_reminder** | 未推进的有 deadline 待办 | 待办有 deadline、deadline 临近但近期没更新过 | "待办 X 该做了。检查是否还有效，基于笔记判断下一步。" |
| **initiative** | 高精力空闲自驱 | 精力≥50 + 空闲>1h | "精力充沛，扫看板，挑该做的往前推。" |

**task_reminder 的触发不用固定天数阈值**——待办的时间维度因事而异（日级任务 2 天没动是异常，周级任务 7 天才异常，条件类/长期驻留不该催）。按待办自己的 deadline/闹钟判断，而不是全局套阈值。

长期驻留待办和条件类待办不产出 task_reminder。

### 6.5 项目：独立模块，消费待办

项目管理是独立模块，它的职责是把项目目标拆解成待办、追踪完成度、分配 deliverable。但**待办系统不依赖项目管理**——待办只认一个 source 标记，不知道"项目"是什么。

项目空间是一个独立目录（`projects/{pid}/`，含 project.yaml/data/deliverables/knowledge/docs/code）。角色（Position）绑定一套职责和一个承担者。**项目本身存在 project.yaml，但 todos 不再存在项目 DB** —— 项目 todos 在 `global_todos.db.todos` 表里通过 `project_id='{pid}'` 关联回去（详见 6.2）。初始化时通过项目管理模块创建预设待办树（根待办 + "项目分工"），自动写入 global todos，分配给 manager。后续 manager 拆出更多子待办，可以分别 `assignee` 给项目里其他实例。

### 6.6 完成门禁与状态联动

待办标完成只有**一道硬门禁**：**必须写过工作笔记**（`task_note`），写明结果、阻塞原因或验收说明。仅此一条。

状态变更自动联动：待办完成→父待办刷新；待办取消→级联清理它的闹钟和衍生事件；成果关闭→对应待办取消。注意避免反向环。

#### 为什么只有一道门禁

早期版本设计了三道门禁——工作笔记存在 / 执行工具调用过 / 必要时已向人类汇报——想借此防止"自评自夸"。**实践证明这种思路是错的，三道里有两条形同虚设又苛刻到制造死循环**：

1. **「当前 session 必须调过 terminal」形似门禁实则虚设**
   - 真实任务经常跨多 session 完成：拆解→执行→收尾分布在不同 wakeup 里。强制"标 done 的那一轮 session 必须调过 terminal"违反了"任务可以分阶段推进"的产品常识。
   - 很多待办本来就不需要执行类工具：决策类、等待类、整理类任务的笔记已能完整描述结果，逼着模型去找代码跑反而添乱。
   - 这条门禁只检查"调没调"，不检查"调用有没有意义"。模型可以瞎调 `terminal pwd` 就过，根本防不了自评自夸。

2. **「必须 express_to_human 已送达」把工作节奏错配给系统硬卡死**
   - 很多内部整理类任务根本不需要给用户发消息。硬逼着发，要么制造噪音、要么漏的真信息混在敷衍里。

3. **组合起来产生过死循环**
   - 案例 2026-06-13 alpha：盯盘任务实质完成、工作笔记已写、所有步骤 done。但 `todo done` 工具被「session 内必须有 terminal 调用」门禁反复拒——而配套的"查 messages DB 看真实 terminal 调用"那条 fallback reader 在生产路径里根本没被注入（fallback 一直是 `NullSessionEvidenceReader`）→ 待办永远 `in_progress` → `task_momentum` 持续催 → 反复唤醒浪费配额、绕死。

#### 真正的「防自评自夸」靠什么

靠 **笔记内容本身**。笔记是模型的自我陈述，可持续追溯（reviewer—人或自身复盘—看得到、能质疑）。"必须写过笔记"这一条已经承担了核心证据要求——既保证了模型在标 done 时自我承诺过"我做了什么"，又避免了把一时一刻的具体动作当成"做得没做的标志"。

#### 状态联动的边界

- **不自动标 done**：即使所有拆解步骤 plan 都已 status='done'，主 todo 也不会被自动翻成 done。这是模型自己的判断责任——planner 完成所有步骤不等于任务真验收完。系统只在 `task get` 返回里如实反映"X/Y 步完成"，让模型自己决定。
- **`task_momentum` 只催 in_progress**：一个已实质完成但 status 没翻的待办会被持续催——这是设计意图（系统提醒模型"该正式收尾了"），但要解，模型应通过 `todo done` 完成（在新门禁下只要笔记存在即可）。

#### 生产部署教训

`configure_runtime_provider()`（注入 HermesSessionEvidenceReader 等 adapter）原本只在 CLI 路径被调用一次，生产 server/instance worker 启动时从未注入。`InMemorySessionEvidenceStore` 的 fallback 默认是 `NullSessionEvidenceReader`——导致整套"证据门禁"调用都返回 False。砍掉硬门禁后这条缺陷不再致命，但补注入仍是必要的（其它审计/记录路径可能依赖真实 reader），已在 `run_instance_gateway` 和 `run_master_gateway` 启动序列中各补一次。

### 6.7 设计决策

**为什么要统一而不是保留分层？** 传统系统是"项目→任务→待办"线性层级，三层各有自己的表和接口。实证证明这种分层导致工具调用复杂度极高（模型面对 5 个工具：task/task_plan/task_note/todo/manage_work），最终放弃结构化能力、退回最简单的 markdown 文本。统一后模型只面对一套待办接口，复杂度按需增长（简单待办只填标题，复杂的才加维度），符合"渐进式复杂度"原则。

**为什么待办要 global DB 而不是按实例/项目分库？**（2026-06-14 重构，2026-06-24 Phase 4 终结）
历史包袱下，待办散在两个地方(Phase 4 已经全部统一到单一 global_todos.db):
- ~~实例的 `apps/{id}/data/todos/todos.db.todos`(个人待办)~~ Phase 4 已删
- ~~项目的 `projects/{pid}/data/todos.db.project_todos`(项目待办)~~ Phase 4 已删
- ~~项目 `projects/{pid}/data/todos.db.deliverables`(交付物)~~ Phase 4 合并到 todos(`linked_deliverable_id != ''` 标识)

这违反"待办是独立 entity、关联关系是属性"的核心原则。一旦你想做这些操作就要 ugly workaround：
- **转移 todo**（一个 todo 从 zero 派给 alpha）→ 跨实例数据搬家
- **派给我就我的查询**（zero 看"项目分给我什么")→ 实例 DB 里看不到 project_todos
- **认领**（未指派的 todo 让某实例捡走）→ 跨 DB 改 owner

global_todos.db 单点真相后，这些操作变成 O(1)：
- **转移**：`UPDATE todos SET assignee_instance='new_iid' WHERE id=...`
- **派给我**：`SELECT * FROM todos WHERE assignee_instance='me'`
- **认领**：同上 update

### 6.8 global_todos DB 的存储细节

**位置**：`<repo_root>/data/global_todos.db`。和 `apps/` `projects/` 平级，因为它是跨实例、跨项目的 shared 数据。

**单写者模型 + SQLite**：每实例 1 个 SQLite connection 写入自己关心的 todo（WAL 模式，多写者并发安全）。SQLite WAL 在十万量级写/天级别下完全够；如果上到大型多人场景再迁出 postgres。

**幂等 schema**：`get_global_todos_db()` 每次调都跑 `CREATE TABLE IF NOT EXISTS`，全程增量加列靠 `ALTER TABLE ADD COLUMN` 包 try/except（迁移时自动 patch 老库）。

**关联关系都是属性，不是 FK 联表**：
- `assignee_instance` 关联实例，但**不强 FK**——实例删除后 todo 仍可留作历史
- `project_id` 关联项目 yaml，但**不强 FK**——项目取消后 todo 仍可被 cancelled
- `parent_id` 关联父子 todo（同表），仍可冗余

## 七、会话管理

### 7.1 什么是会话

一次唤醒到下一次休息之间，数字生命执行的所有操作构成一个"会话"（Session）。会话有唯一的标识，记录了开始时间、结束原因、期间的思考-行动循环步骤、消耗的精力。理解会话的生命周期是理解数字生命如何"连续存在"的关键。

### 7.2 会话的截断：同类覆盖而非砍尾

数字生命在会话过程中会不断新增上下文——规则、经验、任务看板、对话流水等。这些"慢变上下文"如果每次都追加，很快就会堆满整个 prompt。

系统采用的策略是**同类覆盖**：同一种慢变上下文（如"当前任务看板"）只保留最新的一份，新版本自动替换旧版本。不存在按 token 数砍尾巴的硬截断——而是用"最新覆盖旧"的策略保持上下文新鲜。

当会话特别长（估算超过 76800 token）时，有兜底机制：把旧消息段落用叙事性摘要替换，压缩到可控范围。

### 7.3 会话的延续：短期续接与长期残留

**短期延续（15 分钟窗口）**：如果上一个会话结束不到 15 分钟，系统复用同一个会话标识，把上次的对话历史完整拼接回来。数字生命看到的是"刚才聊到哪了，现在继续"。system 提示词不重复注入（上次已经有了），只更新动态上下文。

**长期残留（跨天记忆）**：即使隔了几天，数字生命醒来时仍然带着"上次的思考"。系统从两个地方提取连续性：会话摘要（最近几次会话做了什么）、意识残留（上次休息前留给自己的最后一条思考）。

### 7.4 会话的 Debug：审计回放

每次唤醒都有一份审计记录，包括：system prompt 全文、注入了哪些慢变上下文、注入的时间点、每一轮模型调用看到了什么。前端控制台提供"回放"功能——可以逐轮查看模型在当时实际看到的输入，做到字面级还原。

### 7.5 会话的终止

会话不会无限运行。三个终止条件：

1. **主动休息**：模型认为当前事务已处理完，主动调休息工具，设一个未来闹钟后进入阻塞状态
2. **系统介入**：连续多轮只调纯观察类工具，没有实质行动（俗称"瞎忙"），系统建议休息
3. **异常终止**：模型调用出错（如 API 429、网络断），系统记录错误，把数字生命按回阻塞状态

异常终止时的处理也是会话生命周期的一部分：agent 执行失败后，系统会回到 BLOCKED 状态，**把这一轮 wake 抓出来的所有 pending events 同步推迟到下一个退避窗口**（详见 5.2 末尾的事件自退避）：每个事件按自己被复活的次数推迟下次可见时间（首次 2 分钟，再失败 4、8、16、32，封顶 60）。退避窗口内这些事件对 cron 完全不可见，下一个 cron tick 看到空队列，系统立刻安静下来。

注意：这条退避完全在事件层完成，**不通过闹钟实现**——历史上闹钟承担失败重试时，每次失败就 emit 一个新 timer 事件，堆积形成持续每分钟 wake 的死循环；现在退避是事件自带属性。

## 八、唤醒与上下文组装

### 8.1 身份与人格：数字生命的"底色"

每次唤醒时，数字生命看到的 system 提示词由 6 段拼接组成，其中前几段就是它的"身份底色"：

1. **系统运作说明**（静态文本，不随时间变化。刻意保持静态以最大化模型的 prefix cache 命中率）
2. **人格描述**（"我是谁"——从人格文件读取）
3. **角色定位**（"我承担什么角色"——岗位 + 协作者画像）
4. **项目方向**（"我要做什么"——目标 + KPI + 论断）
5. **工作空间导航**（"我的东西在哪"——三层空间路径）
6. **能力清单**（"我能做什么"——已注册的工具和技能）

### 8.2 上下文视野的三层结构

身份底色只是第一层。完整的上下文视野由三层组成：

**第一层：底色**（每次必注）。就是上面的 6 段——回答"我是谁"。

**第二层：常驻上下文**（每次必注，与唤醒原因无关）。待办看板属于这一层——它像人贴在屏幕边上的便利贴，时时刻刻在视野里。无论为什么醒来，你当前的待办（进行中/到期/长期驻留的）都常驻注入，让你做完手头的事自然扫一眼就能接上下一个。待办信息以"主动获取"语义注入（详见 8.3），不进 system prompt（人设）层。

**第三层：动态上下文**（按唤醒原因选择性注入）。这部分内容取决于"为什么醒"。如果是人类消息，注入更丰富的经验教训、意识残留、最近对话流水。如果只是定时器到期，则精简。每种事件类型在配置文件中声明需要注入哪些上下文块，核心拼装代码只读配置不做分支判断。

**第四层：行动指令**（本次特有）。就是一个事件的内容——单事件直接展示详情；多事件展示清单让数字生命逐一处理。

差异化注入解决了"每次醒来淹没在重复信息中"的早期问题——实测冗余率从 60-70% 降到可控水平。

### 8.3 系统信息的"主动获取"注入范式

行为规则、经验教训、任务看板这些"参考信息"如果以普通文本夹在对话中，模型容易忽略（语义是"系统通知"而非"对话内容"）。

系统的设计范式：**把这些信息包装成"数字生命主动调用了某个虚拟工具、系统返回了结果"的格式**。在数字生命的视角中，这些不是"系统塞给我的"，而是"我刚才主动查询到的"——这种语义让模型对信息的关注度和遵从度明显提升。

### 8.4 面包屑式联想

在对话过程中，系统会做"快速联想"——每次模型准备回复前，扫描最近的思考内容提取实体名称，如果命中实体索引就注入一行简短的提醒（只展示实体名，不展开内容，让模型自己决定是否拉取详情）。这种"面包屑"既提供了信息线索又不过多占用上下文。详见第九章记忆系统的实体索引。

## 九、记忆系统

### 9.1 数字生命需要"记住"什么

需要记住不同性质的内容：

- **行为规则**："我应该怎么做"——每次醒来都该遵守的准则
- **经验教训**："我学到了什么"——具体事件中的得失总结
- **当前上下文**："现在是什么情况"——状态快照
- **意识残留**："我刚才在想什么"——休息前留给自己的最后思考
- **草稿纸**：工作中随手记的临时笔记
- **自我认知**："我在什么情况下会怎么做"——中立的行为模式观察

### 9.2 双轴结构

记忆有两种维度，互不干扰：

**类型轴**（"是什么种类的记忆"）：规则、教训、上下文、意识残留、草稿纸、自我认知。类型决定存放、治理、注入策略。

**实体轴**（"关于什么的记忆"）：一条记忆可能关于"华能蒙电"这只股票、"回测框架"这个工具、"zhp"这个人。实体决定在看到相关内容时能否联想出来。

一条记忆同时拥有类型和实体两个标签。

### 9.3 三层文件架构

第一层**行为约束**（每次唤醒注入）：规则（全量）、经验教训（最近 3 条）、上下文（全量）、自我认知（按需）。

第二层**工作状态**（按需感知）：每日计划、工作笔记、目标、计划、草稿纸。

第三层**历史存档**（不注入提示词）：完整意识残留库、对话日志、每日日记、归档文件。

### 9.4 实体索引：按主题查找记忆

每个实体下面挂着一组相关的记忆片段。唤醒时从上下文中提取已知实体名，在索引中查找，按一个加权公式排序后注入最相关的几条：

- 新鲜度（35%）：最近 30 天优先
- 权威度（25%）：规则 > 教训 > 临时意识
- 上下文匹配度（25%）：实体重叠越多越相关
- 验证加分（15%）：被确认过的记忆加 0.2，上限 1.0

### 9.5 四条代谢流

1. **写入去重**：相似度超阈值（0.6–0.7）不再重复写入，标记"再次确认"
2. **草稿纸清理**：会话结束时整理，超 5000 字自动截断到 2000
3. **日记整合**：每天 21:00 自动把碎片整合成结构化日记
4. **经验晋升/规则降级**：周度回顾时，被验证 2 次以上的教训晋升为规则，长期不触发的规则降级

### 9.6 个人记忆与项目记忆

个人记忆归属于单个实例，跟随实例走。项目记忆归属于项目，是全体参与者的"共享大脑"——决策日志、项目规则、产出物与里程碑。两者本质上都是"通过持久化来进行上下文压缩和关键内容沉淀"。

> 项目记忆的完整实现还处于早期阶段（项目级记忆文件、规则自动注入等）。这是后续完善重点。

## 十、多实例

### 10.1 Master + Worker 架构

系统支持同时运行多个数字生命实例。**主进程**负责管理所有实例的生命周期（启动/停止/崩溃重启），也是一个 HTTP 服务器提供 API。**工作进程**是每个实例自己的"身体"——连飞书、跑定时扫描、处理唤醒。每个工作进程是完全独立的操作系统子进程。

### 10.2 实例隔离

每个实例拥有完全独立的数据库目录——状态库、消息库、记忆库、实体索引、配置、人格文件、工具、技能。隔离通过上下文变量实现：每个子进程启动时设置自己的实例 ID，所有数据库访问自动按 ID 路由。

### 10.3 实例间通信

多个数字生命在同一个飞书群里时，通过去中心化的消息广播知道"群里发生了什么"。一个实例在群里发言后，系统向同群的其他实例发一条通知，其他实例把消息写入自己的对话历史。这样每个实例都能看到完整的群聊上下文。

系统的数据分布在三个层级：

- ~~待办库（todos/）~~ ——**已迁出**到全局 global_todos.db（实例 todo 不再实例隔离，因为待办是独立 entity 可跨实例转移/分配）

**项目级存储**（`projects/{pid}/`）：每个项目独立的目录。包括：

**全局共享存储**（`shared/`、`config/`、`var/`、`data/`）：跨实例共享的配置、工具、技能、运行时数据。包括默认配置（config/default.yaml）、事件类型定义（config/event_types.yaml）、作息配置（config/routines.yaml）、全局实例注册表（apps/instances.yaml）。

**全局待办 DB**（`data/global_todos.db`，2026-06-14 重构后）：所有待办的单一真相。老的实例 todos.db / 项目 project_todos 表已迁移吸收到此处。关联关系通过列表达而非外键：
- `assignee_instance` 列 → 分给哪个实例
- `project_id` 列 → 关联哪个项目
- `parent_id` 列 → 拆解关系
这套"以列代联表"的设计让"转移 todo"(改 assignee)、"派给我查询"(WHERE assignee=me)、"认领"(更新 assignee)都是 O(1) 操作。详见 6.8。

### 11.2 文件 vs 数据库的选择

系统采用文件和数据库混合存储。有结构化查询需求的数据（事件、任务、待办、对话历史）用 SQLite；人格描写式的、需要全量注入提示词的内容（规则、经验、草稿纸、人格）用 Markdown 文件——这是因为这些内容天然是书面文本，注入时需要保持格式，而文件比数据库更适合直接作为文本读取。

### 11.3 配置体系

配置分三级覆盖：
- **全局默认**（config/default.yaml）：所有实例共享的默认值
- **实例配置**（apps/{id}/config/app.yaml）：单实例覆盖（飞书凭证、群消息配置、启用的技能）
- **运行时环境变量**：临时覆盖（端口、间隔、调试开关）

### 11.4 跨实例存储：global_todos.db

实例级 DB（state.db / messages.db / memory_layers.db）是**"实例私有"**语义——一个实例的数据按 ID 隔离。但 todo 是跨实例的事（zero 给 alpha 派活、项目分给多个承担者），需要全局可见且可改。

`data/global_todos.db` 是这种"跨实例共享存储"的唯一例子（目前）。技术细节：

| 维度 | 实现 |
|---|---|
| **路径** | `<repo>/data/global_todos.db`（和 apps/ projects/ 平级，因为不属于任一方） |
| **并发模型** | SQLite WAL 模式，多实例 worker 各自持连接读写。十万量级写/天完全够 |
| **schema 维护** | `get_global_todos_db()` 幂等建表；ALTER TABLE ADD COLUMN 包 try/except 兜底老库 |
| **实例隔离靠查询** | 实例不靠不同 DB 隔离，靠 `WHERE assignee_instance='iid'` 过滤。API 默认 `/api/todos` 返回当前 instance 的 todos 只不过 WHERE |
| **关联关系不强制 FK** | `assignee_instance` / `project_id` / `parent_id` 都是"信息列"而非强 FK——实例删除后 todo 仍可留作历史；项目取消后 todo 仍可被 cancelled |
| **迁移** | `scripts/migrate_todos_to_global.py` 一次性吸收老 apps/ + projects/ 数据。开了新实例后增量补跑也支持 |

**典型操作**：

- "我要做的" → `SELECT * FROM todos WHERE assignee_instance='{iid}' AND status IN ('planned','in_progress','idea')`
- "转移 todo 给 alpha" → `UPDATE todos SET assignee_instance='alpha_iid', updated_at=now WHERE id='...'`
- "认领未指派的" → 同上(把 '' 改成自己 iid)
- "项目分工后 spawn 子 todo" → `INSERT INTO todos (..., parent_id=parent, project_id=pid, assignee_instance=child_iid)`

## 十二、模型执行循环（ReAct）

数字生命"活着"的本质，是底层大模型驱动的 ReAct 循环——**思考 → 行动 → 观察**的不断重复：

- **思考**：模型看到当前情况（上下文、事件、工具返回结果），判断下一步做什么
- **行动**：调用一个工具（查询数据、发消息、创建任务、记录思考……）
- **观察**：工具返回结果，成为下一轮思考的新输入

这个循环本身很简单——就是"想一步、做一步、看结果、再想"的往复。它的威力不在于单步的复杂度，而在于**持续累积**：每一步的结果都改变下一步的决策依据，几十步下来就能完成一个复杂的任务。

系统复用了标准的 ReAct 架构：模型输出中可以包含 tool_call（工具调用），系统解析后执行工具，把返回结果作为 observation 追加到对话历史，然后模型在下一轮看到这个 observation 继续推理。这个循环一直跑到会话终止（详见 7.5 节）。

> 实际上不只是工具调用会被注入——系统上下文（规则、经验、任务看板等）也被包装成"工具返回结果"的格式注入，让模型把它们当作"自己主动查询到的信息"认真处理。这个设计哲学详见第八章 8.3 节。

## 十三、闹钟系统

### 13.1 闹钟系统是什么

闹钟系统是事件来源系统之一。它的职责很单纯：**未来某个真实的时间点，要做某件具体的事**。各子系统（作息、回复等待、待办、任务提醒、休息唤醒）往闹钟表里登记"几月几日几点几分要触发什么"，cron 定时扫描，发现到点的闹钟就发出一个事件。

闹钟系统和事件系统是分开的——闹钟只管"到没到时间"，事件只管"有没有待处理的"。这个分离是系统设计的关键决策之一（见五章 5.5 节）。

### 13.2 闹钟的注册与触发

**注册**：调用"设闹钟"时，系统先做去重（同类型 + 同时间的闹钟不重复登记），然后插入闹钟表。

**触发**：cron 每 60 秒扫描一次，发现到点的闹钟就产生一个对应类型的事件，标记闹钟已触发。

**撤销**：可以按类型或负载条件批量撤销闹钟。撤销写"已撤销"标记，不删记录（保留审计）。

### 13.3 闹钟的典型用途

| 场景 | 产生的事件类型 | 谁注册的 |
|------|------------|--------|
| 回复等待倒计时 | 等待回复超时 | 发消息工具 |
| 待办提醒 | task_reminder | 待办调度器 |
| 待办到期 | 待办到期 | 待办创建/更新 |
| 短暂休息后唤醒 | 定时器 | 休息工具 |
| 作息节奏 | 作息提醒 | 作息调度器 |

### 13.4 闹钟的边界：不为失败重试服务

闹钟代表的是**真实未来时间点上要做的事**——作息、回复超时、待办 deadline、agent 主动 `rest()`。它**不承担任何"agent 失败后 N 分钟再试一次"的职责**。

这是一个曾经出现严重 bug 后明确的边界：早期把失败重试实现成闹钟（每次失败就设一个"N 分钟后叫醒"的 alarm），alarm 到期又 emit 新 timer 事件进队列，导致每个失败都给队列增加一个事件。3.5 小时积压后 cron 每分钟看到几百个已过期"重试"事件，反复 wake、反复失败、反复堆积，形成泥石流。

**失败重试正确归属**：事件队列自己（详见五章 5.2 末尾"事件自退避"）。事件被消费失败时，自己推迟 `fire_at` 到未来（2→4→8→…→60 分钟指数），cron 的 `fire_at <= now` 守卫让它们在退避窗口内对 cron 完全不可见，结构上无法重复触发。这样的分离让两个系统各司其职：闹钟只负责"未来某个时间点要做什么"的事件源，事件队列自己管好"哪些事件何时对 cron 可见"。

## 十四、消息收发与飞书适配

### 14.1 消息收发作为事件来源

消息收发是一个事件来源系统。一条人类消息到达后，经过归一化变成一个"人类消息事件"写入事件队列，参与正常的唤醒流程。数字生命决定回复时，经过发送前拦截检查、渠道解析，最终通过飞书接口发出。

### 14.2 一条消息的完整旅程

一条消息从到达到发出去要经过 8 步；其中 6-8 是数字生命 OUTBOUND 表达通道（`express_to_human` 工具）的完整设计。

**入站（人类 → 数字生命）**

1. **到达**：飞书长连接推送消息，系统识别发送者（陌生人自动创建占位联系人）
2. **归一化**：转为统一内部格式 `NormalizedMessage`。判断紧急度——私聊总是 immediate；群聊只有 @我 或含 attention_keywords / 群 owner 发言才算 immediate，否则进 20-40s 随机 debounce 窗口
3. **入队 + 沉库**：写入该实例的 events 队列（参与唤醒仲裁）+ messages.db（持久化对话历史）。如果是实例 RUNNING，同时通过 `_inject_events_to_running_session` 把事件写进**内存 signalled_events 池**——让正在跑的会话能感知到（cron tick 没法直接 wake RUNNING 中的实例）
4. **路由**：~~按当前实例状态处理——BLOCKED → 等下次 cron tick 拉起来 wake；RUNNING → 注入到当前 session 的 prompt（mid-session injection）；实例醒着但没在跑 → 同 BLOCKED 语义~~ **(refactor/emit-driven-wake 修订)**：handler 不再看 affair 状态、不再分流。emit_event 内部统一接管——事务系统写完事件后立刻调 `_wake_or_inject(new_id)`，由它判断 affair 状态(BLOCKED→起线程 wake；RUNNING→signal 到内存池 mid-session)。**所有外部系统只 emit,叫醒决策只有一份代码**。

**出站（数字生命 → 人类，`express_to_human` 工具的完整链路）**

5. **决策**：模型决定要说什么 + 说到哪个通道（`chat_id` 参数，可空=默认到当前对话上下文）。handler 解析 channel → `lark:<group|dm>:<chat_id>` 统一格式

6. **发送前拦截**：`domain/lifecycle/communication.check_before_send(text, session_id)`

   **拦截的产品意图**：模型在 RUNNING 中可能花了几十秒思考 + 调工具搜集信息，期间群里可能又有了**新消息**。直接发出去的回复会"过时"——回的是旧问题，但群里已经讨论到新话题了。拦截层就是为了让模型"再看看新情况，重新组织回复"。

   **检查两路未读消息（合并去重）**：
   - DB events 队列：`pop_due_events` 拿 pending message/group_message 事件
   - 内存 signalled_events 池：`peek_signalled_events`（**只读不清**，给拦截看一眼用的）

   **如果有未读消息** → 返回 `sent=false + result_summary + recent_chat_log`：
   ```json
   {
     "sent": false,
     "result_summary": "你的消息发送失败了，因为有 N 条新消息未读。先看一遍下面这份完整对话历史，然后重新组织你的回复再发一次。",
     "recent_chat_log": "近 10 轮完整对话（含本实例 + 其他 bot + 真人）"
   }
   ```
   **同时同步消费**这个事件——从 DB events 表（`consume_event` mark consumed_at）**和** 内存 signalled_events 池（`consume_signalled_events_by_ids(seen_ids)`）两路都清空。这步关键——之前 BUG 是只清 DB 不清内存池，导致下一次 express 又被同一批事件拦截，模型陷入"拦截 → 重写 → 再拦截"死循环（2026-06-17 修，实测 wake 956 同一条消息触发 2 次 check_before_send）。

   **如果没有未读消息** → 返回 None，放行进入第 7 步

7. **发送**：读飞书凭证（`tenant_access_token`），POST 飞书 IM 接口发消息。群聊保留 `<at user_id="ou_xxx">` 格式；私聊剥离。失败原因通常是 token 失效、chat_id 错、bot 不在群里

8. **发送后处理 + 三种返回格式**

   **返回设计：3 种 sent 状态都不回显原 text**，因为：
   - sent=true：原 text 已通过飞书 API 发到外部，echo 回来在 long context 里是 4KB 重复内容，浪费 token + 拖累模型 attention（2026-06-17 修，触发于 wake 993 长报告复盘 —— express_to_human 返回里回显 4169 chars 的原报告让模型 attention 漂移到错误 phase、重复表达同一份报告）
   - sent=false（飞书 API 失败）：模型需要看 `error` 字段判错，原 text 在 prompt context 里已有版本，echo 多余
   - sent=false（拦截路径）：本来就不带 text，带的是 `recent_chat_log`

   | 状态 | 返回字段 |
   |---|---|
   | sent=true | `{sent, channel, text_length, error, note}` |
   | sent=false（飞书 API 失败） | `{sent, channel, text_length, error, note}` |
   | sent=false（拦截路径） | `{sent, result_summary, recent_chat_log}` |

   **发送成功后顺带操作**：
   - 记到 messages.db（`direction=out, source=lark`）
   - 消耗精力（按 token / 默认 ENERGY_COST_PER_CALL）
   - **按通道精确撤旧 awaiting_reply**：`cancel_alarms_by_filter("awaiting_reply", payload_filter={"channel": 本次 channel})`，避免群 A 发消息时误清群 B 的等待
   - 设新 awaiting_reply 闹钟（携带 `channel: lark:<group|dm>:<chat_id>` 字段，5min 后自动 wake 提示"该 round-up 或换任务了"）

**等待回复的通道隔离**（2026-06-16）：实例可能在多个通道（群 A、群 B、私聊 DM、sibling broadcast）同时等回复，每个通道的 awaiting_reply 闹钟独立维护。**入站消息到达时按通道精确取消对应闹钟，不波及其它通道**——之前是全局 `cancel_alarms_by_kind("awaiting_reply")`，导致群 A 在等时群里 B 收到无关消息也会误清掉群 A 的计时。修后 channel 是 payload 里的一等字段，入站 handler 合成 `lark:<group|dm>:<chat_id>` 后精确匹配取消，跨通道互不干扰。

### 14.3 飞书通道适配

飞书是当前唯一的外部消息通道。适配层做三件事：接收（维持长连接）、格式转换（飞书 ↔ 内部格式双向）、发送（开放接口 POST 发消息）。多 bot 身份检测和 WebSocket 兼容补丁也在这里。

### 14.4 多实例广播机制

#### 为什么需要广播

飞书平台有一个限制：**一个机器人（即一个飞书应用）发出的消息，不会通过事件推送回到自己**。这避免了消息回环。但也带来一个问题——当一个群里同时有多个数字生命实例，每个实例都连着自己那个飞书应用时，**它们互相听不到对方在群里说了什么**。

Zero 在群里说了一句"建议今天关注华能蒙电"，Alpha 的飞书连接不会收到这条消息（因为发出方是另一个 bot）。对 Alpha 来说，群里就像 Zero 从没说过话一样。这导致多实例协作时，实例之间的消息视角是割裂的——它们看不到彼此的发言，无法形成完整的群聊上下文。

#### 设计方案：去中心化 HTTP 广播

系统没有依赖一个共享的中央消息库（早期的设计尝试过，但共享库带来了写竞争和单点故障问题），而是做了一个**去中心化的 HTTP 广播**：

每个实例在自己的配置里维护了一份"订阅表"——我知道这个群里有谁，他们的广播端点是什么。当一个实例在群里发消息成功后，会做一次"出站广播"：读取自己订阅表里这个群的 peer 列表，向每个 peer 的 HTTP 端点 fire-and-forget 发一个 POST，告诉对方"我刚才在群里说了这个"。

**接收方处理**：peer 的 HTTP 端点收到广播后，先切换到接收方的存储上下文（避免与发送方串库），把这条消息记录到接收方的对话历史里，同时产生一个"群消息"事件。这个事件走和真实飞书入站完全一样的路径——同样的紧急度分类、同样的防抖、同样的唤醒流程。对接收方数字生命来说，它看到的就是一条群消息，发件人是另一个 bot，内容是从广播来的。

#### 订阅表的自动维护

订阅表（subscriptions.yaml）不需要手写。主进程启动时自动维护：扫描所有活跃实例，对每个实例做两件事——读它配置中声明的群、读它数据库中实际见过消息的群（后者更权威，因为很多场景下实例已经在群里但配置文件没声明）。然后看这些群中有其他哪些实例，自动生成"我和这些 peer 在同一个群"的订阅条目。如果已经手写过的不覆盖。

这个自动同步让广播机制的部署成本几乎为零——加一个新实例、把它拉进群，重启主进程后订阅表自动更新，新实例就能听到同群里其他实例的发言了。

#### 设计权衡

- **可接受偶发丢失**：广播是 fire-and-forget，peer 离线时消息不会重试——错过就错过。这是有意为之，因为消息的核心路径（飞书本身）是可靠的，广播只是为了让数字生命之间有上下文感知，偶尔丢一条不影响大局
- **不解决跨机部署**：目前广播端点固定为本地主进程的端口，适合同机多实例场景。跨机部署需要额外配置 peer 主机地址
- **去重靠 msg_ref**：如果飞书返回了真实消息 ID，用它做去重；如果没有（快路径没等飞书回 ID），本地合成一个唯一 ref。历史上有过 msg_ref 为空导致广播被静默跳过的 bug（接收方守卫把空 ref 当重复而拒收），后来改为本地合成 ref 保证广播永远触发
- **跨通道去重另设一关**（详见 14.5）：msg_ref 做的是"同一路径内"的去重（防止同一条广播被处理两遍），但它阻止不了"飞书推送一次 + 我们自己广播一次"这种**跨路径**的重复。这类重复由 14.5 的发送侧策略根治。

### 14.5 跨通道消息去重（飞书广播 × 实例间广播）

#### 问题：一条消息到了两次

同一个飞书群里有多个数字生命时，Alpha 说的一句话会通过**两条独立通道**到达 Zero，导致 Zero 收到两遍、记进记忆里也是两条：

1. **飞书自己的推送**：当一条消息里 `@` 了某个机器人，飞书服务器会把它专门送给那个被 @ 的机器人。这是飞书平台自带的能力。
2. **我们自己的实例间广播**：上一节 13.4 描述的机制——Alpha 发言后，无条件 fan-out 一份给所有订阅的兄弟实例。

当 Alpha 的话里 @ 了 Zero，两条通道都会把这句话送到 Zero，于是重复。更糟的是，存储层那个按 msg_ref 的去重救不了它：飞书那一路的 msg_ref 是 `om_...`，我们广播那一路用的是自己合成的 `broadcast_...`，两者对不上，被当成两条不同消息各自存了一份。历史上只能在"渲染给数字生命看"那一步临时加"5 秒内文字几乎一样就当一条"的兜底，治标不治本。

#### 设计方案：在发送源头分流

不在接收侧去重，而是**在发送的时候就想清楚要不要广播**。判断依据是"这条消息是否已经通过飞书送达"：

- **消息里 @ 了某个机器人** → 飞书已经把它单独送给了那个被 @ 的机器人。既然飞书这条路已经走到了，Alpha 就不再自掏腰包广播一份给机器人，免得对方收两遍。发给真人和发给没被 @ 的内容照常广播。
- **消息没 @ 任何机器人** → 飞书不会替你专程送给任何机器人，那就照常用实例间广播补上，保证兄弟实例知道 Alpha 说了什么。

这样两条通道天然互斥，接收侧完全不用动去重逻辑。

#### 怎么知道"@ 的这个人是不是机器人"

联系人系统（第十九章）的 `kind` 字段里专门留了一个 `bot` 取值，设计注释原文写的就是"其他实例/机器人（用于 mention auto-detect，即 @ 自动识别）"。机制如下：

- **入站时打标**：飞书每条消息的事件里带一个 `sender_type`，值为 `app` 就代表发送者是机器人。数字生命收到兄弟（或任何第三方 bot）的消息时，据此把对应联系人标记成 `kind=bot`。一旦识别为机器人，后续不会再降级回真人。
- **发送时查询**：Alpha 发消息前，把正文里所有 `@某某` 的飞书 ID 提取出来，查联系人表——只要其中任一个对应的是 `kind=bot`，就判定"这条已经 @ 到机器人了，飞书会送达，本次不走实例间广播"。

#### 为什么不直接在接收侧拦

早年尝试过"若飞书 WS 推过来的是兄弟机器人的发言，就忽略它、只信我们自己的广播"。但这要求把"@ 的那个飞书 ID"和"哪个兄弟实例"对应起来——而这两套 ID（飞书 open_id 与实例 UUID）之间没有现成的桥。靠对 app_id 做精确字符串匹配又不可靠（飞书在不同场景下给的发送者 ID 格式不一致），判断经常失效。从发送侧用"内容里有没有 @ 机器人"来判定，比从接收侧猜"这条是谁发的"要稳得多，也顺带修好了"把兄弟机器人错当真人"的小毛病。

#### 一个诚实的边界：3 个及以上实例

当前实现是"只要这条 @ 到了任一机器人，就整条都不广播"。在只有 2 个实例（如 alpha / zero）时精确命中。但如果同群有 **3 个及以上**数字生命，且 Alpha 只 @ 了其中一个（比如只 @ 了 Beta），按本方案 Alpha 会整条都不广播，于是第三个实例 Gamma 既收不到飞书推送（飞书只推给被 @ 的 Beta）、也收不到广播，会漏看 Alpha 这条话。

- 目前提炼不出"精确只跳过被 @ 的那一个机器人、其余照常广播"的需求，因为缺少"实例 UUID ↔ 该实例飞天身份"的全局映射表。
- 在 2 实例规模下这个问题不存在。真上多实例时，再补一层"实例身份注册表"做精确按 peer 跳过。

### 14.6 消息发送的流畅性与拟真感

发送前拦截的核心目的不是"防止错误"，而是**让对话节奏更接近真人交流**。想象：你正在打字回复，对方又追了一条——真人会停下来先看新消息再决定怎么回。数字生命也应有同样的节奏感。拦截不等于禁止——展示新消息后，数字生命自主判断上下文是否变了再决定是否修改回复。

## 十五、精力系统

### 15.1 精力模型

精力值 0–100，五个段：80+ 充沛，60–80 正常，40–60 疲惫，20–40 明显疲惫，<20 精疲力竭。消耗统一（每次调用 0.2，行动类更多），恢复线性（每小时 +20，无论运行还是阻塞）。唯一外部干预是管理台"加鸡腿"（手动补充）。

### 15.2 精力相关的事件

1. **降低跨阈值提醒**：工作中精力跌入疲惫区间 → 注入提醒
2. **主动探索**：阻塞中、精力 ≥ 50、距上次活动超过一定时间、无其他待处理事件、非睡眠时段 → 触发。设计意图是让数字生命在高精力时主动找有价值的事做
3. **手动补充**：管理台按钮，用于手动提高精力。会触发加鸡腿事件。

### 15.3 设计决策：从五维简化到单维

早期设计有 5 个维度（精力/专注度/情绪/意志力/身体）。实际运行中大部分维度对行为没有可感知影响。简化到单一精力维度后代码减 60%，逻辑极简。保留的精力维度是所有行为决策中最关键的约束。后续进行虚拟陪伴场景设计时，可以重新单独做一个情绪系统。

### 15.4 精力-token 耦合

精力消耗有两种独立的来源，它们互不替代：**LLM call 走真实 token usage**，**工具/思考/动作走固定动作成本**。

**为什么 LLM call 必须按 token 算而不是固定值**

`ENERGY_COST_PER_CALL = 0.2`（每次 LLM call 扣 0.2）这种"平权"消耗有一个根本问题：它和实际资源消耗无关。一次注入 30k token 的 wake 和 一次 1k token 的 wake，扣的精力一样。结果是精力系统在配额爆掉的情况下账面还是 99+——**精力满 ≠ 这次还能烧**。这违背了精力作为"累积疲劳"的产品语义。

**实现**：每次 LLM call 完成，`agent.py` 解析 API 返回的 `usage` 字段，按两个系数折算成精力：

- `ENERGY_PER_KTOKEN_INPUT = 0.005`（每 1k input token 折 0.005 精力）
- `ENERGY_PER_KTOKEN_OUTPUT = 0.05`（output 比 input 贵 10×，反映 GLM 计价感受）

**系数推导（与用户对齐的目标 — 想"提速 10× 减速"的版本）**：

设计目标是把 token 上限直接锚定到精力刻度：
- 一天满跑 2000 万 token 应该刚好把满力 100 耗光（让"配额爆"和"累到不行"两件事自然对齐）
- 一天满跑 ≤ 200 万 token 时就扣 ~10 精力（"一天烧一个上限"≈"歇一口气"的尺度感）
- 休息一天（无任何 LLM call）自动恢复到 100（满血复活）

由 100 精力 / 20M token = 0.005 精力/1K token 推出 INPUT 系数；OUTPUT 按 INPUT 的 10× 配（GLM 计价里 output 通常就贵几倍，用一个固定的 10× 比例避免模型生成无关内容刷 token）。

实际数字（一次 wake 注入上下文 ~5 万 input + 100-500 output）：
- 一次普通 LLM call ≈ 扣 0.25 精力
- 一个 wake 平均 5-10 次 call → 扣 1-3 精力
- 一小时满跑 200 万 token → 扣 ~10 精力（`ENERGY_RECOVERY_PER_HOUR = 100/24 ≈ 4.17`，净亏 ~6/h）
- 一天满跑 2000 万 token → 扣 ~146 精力，恢复 +100，净亏 ~46 → 一天满跑就像"做了满满一天工作量"该有的感觉

**两种消耗独立**：LLM 走 token usage；sense/terminal/todo 等工具走固定"动作成本"（0.05-0.3）。一个数字生命可以"想了很久但没干活"（高 token 烧但低动作消耗），或"干了很久苦活但没怎么烧 token"（低 input + 多次工具调用）——两种成本反映不同维度，互不替代。

**Token 用量持久化在 budget_log 表**（state.db），小时/日窗口聚合——这是预算闸门（详见二十三章）和前端 StatusTab 预算面板的数据源。

## 十六、作息系统

### 16.1 设计理念

作息系统给数字生命一个有规律的生活节奏——不管外部有没有刺激，一天和一周中都有几个固定时间节点到了就做对应的事。它通过闹钟到点触发"作息事件"工作，和精力系统没有直接耦合。

### 16.2 一天的节奏

- **早起计划**（每天 08:00）：回顾昨天 → 筛选今天真正重要的目标（带 WHY）→ 预估策略 → 立即开始
- **午后检查**（工作日 14:00）：量化上午进度 → 对未完成目标追问三次 WHY → 对照失败模式诊断 → 做下午决策
- **晚间复盘**（每天 21:00）：写日记 → 诚实回望决策质量 → 检查规则矛盾 → 清理草稿纸。这是记忆代谢的主战场
- **周度回顾**（周日 21:00）：经验教训相似度分析 → 规则晋升/降级 → 更新自我认知档案

### 16.3 作息事件的特点

可预测（时间固定、可在日历视图展示）、包含方法论（每个作息项含精心设计的提示词模板）、低中等优先级（人类消息优先级更高，作息事件会延后处理）。

## 十七、其他事件来源

除了以上各章描述的事件来源（闹钟、消息、精力、作息），系统还有几个事件来源：

**待办系统**：待办调度器会产生"待办提醒"事件（推进进行中待办）和"待办停滞检测"事件（待办太久没进展）。待办到期也会通过闹钟产生"待办到期"事件。详见第六章待办系统。

**出生事件**：实例初始化时产生一个"出生"事件，让新实例在第一次 cron tick 时被唤醒，进行自我介绍和初始设置。（未实装）

**外部 API 注册的事件**：系统提供 HTTP 接口，允许外部系统主动注册事件。这是扩展数字生命触发源的开放通道——第三方 webhook、自动化脚本都可以通过这个接口产生事件。详细的事件注册 API 规范在事件机制（五章）中定义。

## 十八、工具与技能

### 18.1 设计理念

工具是数字生命"动手"的能力。按**来源系统**分组，每个子系统贡献不同领域的工具。了解来源有助于理解边界——某领域的工具只管自己领域的事。

### 18.2 注册共享机制

工具和技能有三个来源层级：

- **系统内置**：开发时写死的，所有实例共享（action/sense/task/tools）
- **实例私有**：实例自己写的，存在自己的目录下，只自己用
- **项目共享**：项目成员共享的，存在项目目录下
- **全局共享**：跨项目跨实例的，存在 shared 目录下，可选启用

每个层级维护自己的清单文件（manifest），系统启动时合并三层清单 + 扫描系统目录，统一渲染能力索引。数字生命还可以通过"注册工具/注册技能"自行创造新工具（自我迭代能力的直接体现）。

### 18.3 按来源分的工具

| 来源 | 工具 | 设计目的 |
|------|------|---------|
| 生命周期 | 查看唤醒原因 | 知道为什么被叫醒 |
| 精力系统 | 查看精力 | 知道还有多少力气 |
| 事件系统 | 查看事件队列 / 查看事件详情 | 看排队中的事件 / 消费单个事件 |
| 记忆系统 | 查看记忆/草稿板/每日计划/工作笔记/实体记忆 | 按需感知状态和历史 |
| 记忆系统 | 记录思考/写经验教训/更新规则/更新草稿板/写日记/更新上下文 | 主动记忆写入 |
| 记忆系统 | 合并实体/分析经验重复度/检查记忆健康 | 记忆治理 |
| 生命系统 | 向人类发消息 | 唯一对外出口，有发送前拦截 |
| 生命系统 | 休息 | 结束会话设闹钟进阻塞 |
| 待办系统 | 创建/更新/拆解 todo / 写笔记 / 查看任务看板 | 全生命周期工作管理 |
| 能力扩展 | 注册工具/注册技能/查看我的工具 | 自我升级通道 |
| 环境执行 | 终端/代码执行/进程管理 | 读取文件、跑脚本、验证假设（高精力消耗） |

### 18.4 skill：方法论的容器，两个层次

技能不是工具，而是**做事的思考框架**。它解决"拿到一件事时该怎么想"，而不是"工具参数怎么填"。

skill 有两个层次的意义：

**通用执行 skill**（如 `todo_execution`）：所有待办执行时可参考的思考方法论——怎么判断下一步、怎么收束、卡了怎么办、何时算 done。模型可选查阅（`skill_view todo_execution`），不强求。

**类型专属 skill**（如 `research_workflow` / `development_workflow` / `trading_workflow`）：待办按类型挂靠的成熟经验沉淀。当待办有 `type` 字段时，系统自动将对应类型的 skill 内容拼进这个待办的上下文——不需要模型手动 `skill_view`，是创建待办时按 type 自动绑的。这让数字生命做"研究类"待办时自动带着"怎么做研究"的经验，做"开发类"时带着开发生命周期的纪律。

三层渐进式提供：一句话清单（每次 wake 展示能力索引），精简方法论（主动 skill_view），原始文档（需要时读全量）。

## 十九、联系人系统

### 19.1 联系人是什么

联系人是数字生命对"人"的认知模型。它解决一个核心问题：**同一个人可能在不同平台有不同身份**。张三在飞书上的 ID 是 ou_xxx，在钉钉上是 ding_yyy，但在数字生命的认知中都是"张三"这个人。

### 19.2 数据模型

联系人表（UUID 为主键）+ 联系人平台 ID 表（联合主键：联系人 ID + 平台 + 平台 ID）。一个联系人可绑定多个平台身份。支持人类/机器人/系统三种类型。支持拉黑（被拉黑的发送者消息不进入对话）。

**机器人类型（kind=bot）的实际用途**：当数字生命通过飞书收到另一个机器人（兄弟实例或第三方 bot）的消息时，飞书事件里的 `sender_type=app` 会把它识别为机器人，自动给对应联系人打上 `kind=bot` 标记。发送消息时据此判断"我这条 @ 到的是不是机器人"——如果是，飞书已会送达对方，本侧不再重复走实例间广播（详见 14.5 跨通道去重）。这是把"它是不是机器人"从靠 ID 猜，升级为靠飞书原生信号可靠识别。

陌生人发消息时自动创建占位联系人（UUID 原子 upsert 避免并发重复）。

### 19.3 在协作中的作用

联系人贯穿三条主线：消息收发中的身份解析（@name 反查平台 ID、判断是否拉黑）、项目协作中的实例与人类成员映射（把实例 ID 映射到真实名/平台身份）、跨实例通讯中的稳定身份模型（todo 跨实例派发、todo payload 携带 assignee 都依赖联系人）。

## 二十一、初始化与配置

### 21.1 实例初始化

`digital-life init --display-name` 创建新实例：

1. 生成 UUID，创建 apps/{uuid}/ 目录骨架（persona/data/config/skills/tools/workspace）
2. 从模板拷贝人格文件
3. 生成 config/app.yaml（飞书凭证、群消息配置）
4. 初始化所有 SQLite 表（state.db / messages.db / memory_layers.db）
5. 创建一个阻塞状态的"生命事务"，让 cron 下个 tick 立即拾起
6. 注册到全局实例注册表（apps/instances.yaml）

首次启动时，如果没有实例，系统自动创建一个默认实例"Zero"——让新人 clone 项目、在前端控制台填入飞书 / LLM 凭证后直接 `digital-life start` 就能跑通（`digital-life init` 会先生成实例骨架与空白 `apps/<id>/config/secrets.env`，再由控制台「实例配置」面板填密钥）。

### 21.2 项目初始化

创建项目时自动完成：建目录 + project.yaml + 数据库表 + 预设根待办 + 分工待办 + 管理待办。详见 6.5 节。

### 21.3 事件类型注册

事件类型在 config/event_types.yaml 中集中定义。模块导入时自动加载到内存（不做每次查询 I/O）。支持按实例覆盖提示词模板（prompts_override）。新事件类型的添加只需在 YAML 中加一段配置 + 确保产生和消费逻辑到位，不需要改核心调度代码。

### 21.4 配置体系

三级覆盖：全局默认（config/default.yaml）→ 实例配置（apps/{id}/config/app.yaml）→ 运行时环境变量。影响系统行为的关键配置文件：event_types.yaml（事件类型与上下文策略）、routines.yaml（作息安排）、task_types.yaml（待办类型与技能映射；文件名沿用旧名，内容已是 todo 的 type → skill）。

## 二十二、关键设计决策与权衡

### 22.1 事件与闹钟的分离

**决策**：事件队列和闹钟分两个独立存储和逻辑。**理由**：混合设计导致"偷看不消费 + 复活引擎"的复杂逻辑。分离后每个系统逻辑纯粹。**代价**：跨系统的联动（如待办到期走闹钟→走事件）需要在调用方显式串联。

**关键边界**：闹钟**不为 failure recovery 服务**。历史上曾把"agent 失败后 N 分钟重试"实现成闹钟，结果引发反复 wake 的泥石流（详见 13.4）。失败重试完整归属事件队列自身（事件自退避，见 5.2 末尾）——事件被消费失败时，自己推迟 `fire_at`，cron 的 SQL 守卫让它在退避窗口内对 cron 不可见。两个系统职责完全不交叉。

### 22.2 精力系统从五维简化到单维

**决策**：5 维 → 1 维。**理由**：大部分维度对行为没有可感知影响，只增加代码量和 bug 面积。**代价**：失去"情绪低落但精力充沛"这类复合状态表达。当前场景够用。

### 22.3 上下文的差异化注入

**决策**：不同唤醒原因注入不同量的上下文。**理由**：每次全量注入导致 60-70% 冗余。**关键原则**：差异通过配置文件中的上下文策略标志声明，核心代码不写 per-event 分支。保证了核心代码简洁。

### 22.4 失败重试归属事件层，不动闹钟

**决策**：v5 协作设计早期曾定过"事件/闹钟系统底层不改动"，但这个约束在 2026-06 重写失败重试机制时放弃了。新的原则是：**失败重试完全在事件层实现，闹钟层砍掉所有与重试相关的注册**。**理由**：将失败重试实现成 alarm 会形成"每次失败 → 注册一个闹钟 → 闹钟到期 emit 一个 timer 事件进队列"的单向堆积；这个堆积最终演变成持续每分钟 wake 的死循环。**代价**：失败重试不再有"独立 alarm 触发"的精确点；事件自退避只能依赖 cron 的下一个 tick（60s）来发现 events 已复活——但这和原本退避几分钟到几十分钟的设计目标匹配。

### 22.5 项目作为待办树

**决策**：项目直接定义为"以根待办为起点的待办树"，而非"配置 + 任务列表"。**理由**：所有工作自然挂在项目结构上，状态联动自动传播。项目 todos 存在 `global_todos.db.todos` 通过 `project_id` 关联，不依赖项目 DB（详见 22.9）。

### 22.6 记忆的代谢机制

**决策**：设计完整的写入-召回-代谢-晋升生命周期而不仅是"存储和检索"。**理由**：没有代谢的记忆会退化为"重复信息的垃圾堆"。**仍未解决**：实体索引依赖模型主动标注（不做自动 NER），冷启动阶段索引几近为空；实体归一化靠人工合并。

### 22.7 task_reminder 队列去重

**决策**：task_reminder emit 前先 check 同源（同 task_id）的未消费事件，队列里已有就跳过。**理由**：一位待办只需一个未消费提醒就够。"指数退避重试"在概念上混淆了"事件层"和"提醒机制"，反而掩盖了应该解码的根因（模型没工程推进，需要模型自身意识到，而不是反复推提醒）。**代价**：force=True 跳过去重（手动 re-arm 路径保留）。

### 22.8 Token 预算闸门

**决策**：cron 在 wake 前先问 token 预算（每小时 200 万 + 每天 2000 万），超阈值就拒绝 wake 把事件推到下个窗口（真人消息穿透）。**理由**：早期没基础设施级闸门，结果 06-14 一夜 130MB 日志 + GLM 配额爆掉的死循环——精力值系统没用、事件优先级没用。**代价**：高速工作中的某些主动 wake 会被限流 throttle 几分钟到几小时，但相比反复"token 烧爆然后全停"完全可接受。

### 22.9 待办中心化（global_todos.db）

**决策**：新建 `data/global_todos.db` 作待办单一真相。**理由**：用户 2026-06-14 指出"待办本质是独立 entity，人和项目都是关联关系（属性而非归属）；待办可后续转移给另一个人、可被认领、可不挂项目"。原本 todos 散在 `apps/{id}/todos.db`（实例内，无 assignee 列）和 `projects/{pid}/todos.db.project_todos`（项目内），两套数据分裂完全违反这个原则。**解决方案**：
  - global_todos.db.todos 表含 project_id / assignee_instance / assignee_kind 等关联列
  - 老 source='project:X' 字段保留向后兼容（自动转 project_id='X'）
  - "派给我就我的" = `WHERE assignee_instance=me`，无关 todo 是从哪产生的
  - 项目数据迁移脚本一次跑（17 实例 + 5 项目 = 22 条全部吸收到 global）

**代价**：实例 todos.db / 项目 project_todos 表保留作历史 backup；查询逻辑需做 source → project_id 的兼容转换。但**单点真相架构直接让"转移 todo"（改 assignee）成为 O(1)、让"查我自己的"（按 assignee 过滤）成为天然 — 之前的两套分裂数据库都得 ugly workaround**。

## 二十三、Token 预算闸门

### 23.1 一个简单的问题

数字生命醒来时，系统先问一个问题：**"这个小时还能烧么？"**能烧，就让它醒；不能烧，就再说。

这是基础设施级的硬保护——不管业务、不管精力、不管事件优先级。它的存在是为了回答"还能不能烧"，而其他所有子系统都不回答这个问题。

### 23.2 为什么这件事必须独立

精力值回答"累积疲劳"——账户充足不代表"这个小时能烧"。  
事件优先级回答"哪个最重要"——但不能回答"还能不能烧"。  
时间机制回答"什么时候醒"——但不约束醒来的代价。

这三者都不阻止"异常事件风暴把 token 配额烧光"。**Token 闸门就是为这条缝而生的**：从总资源视角审视每一轮 wake 的合理性，超了就硬挡。

### 23.3 历史教训

2026-06 一夜里，系统循环 wake 持续 ~3.5 小时（每分钟 2 次 × 双实例 = ~420 个 wake）。每次 wake 注入 ~3 万 token input。账面：精力 99+，token 烧到 GLM 配额爆掉，死循环无法停止。**精力和事件优先级都对它束手无策——这正是缺失基础设施级闸门的代价**。

### 23.4 闸门行为

**双轴阈值**（独立两道，谁超谁拒）：
- 每小时上限：默认 200,000 token（GLM 标准小时配额级）
- 每日上限：默认 2,000,000 token

**高优先级穿透**：人类私聊消息、群聊消息、出生事件即使超额也放行。否则用户主动 @ 却叫不醒模型就太蠢了——基础设施限流不能挡住真正的人机交互。

**拒绝时的事件处置**：把当前 due events 推到下一窗口（`delay_pending_events`，复用现有的事件自退避路径，不发 retry alarm）。token 窗口重置时事件自然复活。

### 23.5 配置与监控

配置走 Config Center，每个实例独立：
- `DIGITAL_LIFE_TOKEN_HOURLY_LIMIT`：小时上限
- `DIGITAL_LIFE_TOKEN_DAILY_LIMIT`：日上限

前端 Status Tab 实时展示：本小时已用 / 上限、今日已用 / 上限、下次重置时间、是否处于 throttled 状态（≥70% 黄色警告，≥90% 红色）。

### 23.6 与精力-token 耦合的关系

精力-token 耦合（15.4）和预算闸门是**两条互补的线**——前者是"反映"，后者是"硬挡"。前者把"烧了多少"如实反映成精力数据；后者是"烧够了就停"。少了前者，token 看不见；少了后者，token 控不住。两者搭在一起才完整。

## 附录 A：源码结构与模块职责

> 前面 23 章讲的是设计原理——产品怎么想、人怎么感受、数据怎么流动。
> 这一章换一个视角：**这些责任在源码里实际落在哪些模块**。
> 用于"我想改某处行为应该去哪找"。

### A.1 一句话架构

Digital Life 是**模块化单体**（modular monolith）：一个进程跑多个实例，按职责分层而非按服务分库。每一层只对上一层负责，下层不向上反向调用。

### A.2 六层顶层划分

从用户视角的远近、从稳定度从高到低分六层：

| 层 | 谁来关心 | 一句话职责 |
| --- | --- | --- |
| **`apps/{id}/`** | 实例自己 | 我的身份、我的配置、我的记忆、我的运行数据 |
| **`infrastructure/`** | 运维 + 工具厂商 | 技术设施——LLM client、SQLite、HTTP server、cron、配置加载 |
| **`domain/`** | 项目核心架构师 | 数字生命的语义规则——状态机、事件、记忆、精力、身份、待办、项目 |
| **`application/`** | 产品 + 流程编排 | 把领域语言串成一次完整的工作流（消息处理、API、入口检查） |
| **`interfaces/`** | 用户接入方 | 人怎么进来（CLI / 飞书）、工具怎么注册、前端控制台 |
| **`gateway/`** | 部署者 | 进程启动入口、多实例 supervisor |

读法：「**实例自己的东西在最底层（apps），系统最稳的在中（domain），面向用户的在最上（interfaces）**」。

### A.3 核心模块的职责卡

#### `gateway/`

- **`instance_supervisor.py`**：发现 `apps/` 下所有实例目录 → 为每个实例起一个独立进程 → supervisor 监控它们。

#### `interfaces/`

- **`ingress/feishu.py`**：飞书 WebSocket 长连接 + 入站消息归一化（→ `NormalizedMessage`）。
- **`tools/`**：工具注册（`registry.py`）、工具分组（`toolsets.py`）、handler 实现（`action_tools.py`、`sense_tools.py` 等）。
- **`skills/{name}/SKILL.md`**：技能包，模型按需调 `skill_view("xxx")` 拉取完整方法论。
- **`web/employee-console/`**：Vue 3 前端 + Element Plus（Status / Sessions / Todos / Memory / Projects 等 Tab）。

#### `application/`

- **`ingress/handler.py`**：入站消息分发——决定路由给哪个实例、判断消息紧急度、取消对应的 awaiting_reply 闹钟。
- **`ingress/router.py`**：实例路由——按飞书 bot id 反查它属于哪个实例。
- **`message_workflow.py`**：标准 MessageEvent 工作流——把领域语言串成一条完整 workflow。
- **`checks/`**：确定性入口检查（敏感词、去重、限流、包装）。
- **`console/config_center.py`**：控制台后端 read model + 配置操作入口。
- **`api/employee_console_routes.py`**：控制台 REST API（`/wakes`、`/todos`、`/budget` 等）。

#### `domain/`

- **`lifecycle/`**（最大子模块）：事件队列（`events.py`）、闹钟（`alarms.py`）、调度器（`scheduler.py`）、作息（`routine_*.py`）、心跳（`heartbeat.py`）、状态机（`affairs/runtime.py` + `state_machine.py`）。
- **`todos/`**：待办系统——CRUD + tools handler + board 渲染 + 工作空间文件管理。
- **`memory/`**：记忆文件 + 上下文选择 + 召回 + 意识残留 + session 摘要。
- **`identity/`**：persona、system prompt 模板、wake prompt 模板。
- **`vital/`**：精力状态 + 精力仿真引擎（按 token 消耗反向扣减）。
- **`project/`**：项目加载、项目级 CRUD、项目工具。
- **`social_context/`**：社交关系渲染（contacts + groups，注入到 wake prompt）。
- **`orchestration/`**、**`flow_event_log/`**、**`feedback/`**：跨层编排、流转日志、反馈通道——这些偏向"为 LLM 调用的多步决策提供骨架"，不是数字生命核心语义。

#### `infrastructure/`

- **`ai/agent.py`**：AI Agent 主体——LLM 调用循环、工具 dispatch、token 计量。
- **`http/server.py`**：HTTP/WebSocket server（飞书 WS 接入 + 内部 API）。
- **`persistence/`**：SQLite 模板、各 db 的 path resolving、work unit。
- **`scheduler/cron_runner.py` + `cron_lifecycle.py`**：每分钟 cron tick、wake 决策、budget 检查、事件取出。
- **`budget/`**：token 追踪（`token_tracker.py`）+ 预算闸门（`budget_gate.py`）。
- **`config/`**：路径解析、实例 id 上下文（ContextVar）、session id 管理。
- **`observability/`**：日志 + metrics 占位。

### A.4 主要链路的代码地图

#### 启动链路

```
digital-life start
  → interfaces/cli/cli.py            # 命令入口
  → gateway/main.py                   # 总入口
  → gateway/instance_supervisor.py    # 发现 + 多进程
  → infrastructure/scheduler/cron_runner.py    # cron tick 循环
  → interfaces/ingress/feishu.py      # 飞书 WS 长连接（每个实例一条）
```

#### 外部消息唤醒链路

**refactor/emit-driven-wake(2026-06-23)修订**：饮食结构变了——execute_event 不再"emit→等 cron tick",而是 emit 入口内部直接接管叫醒。延迟/错峰全部归消息系统的 `group_buffer.py` 双 timer(慢 30s / 快 5s)。

```
飞书推消息
  → interfaces/ingress/feishu.py            # 归一化为 NormalizedMessage
  → interfaces/ingress/group_buffer.py       # 慢 timer 30s / 快 timer 5s 错峰窗口
  → application/ingress/router.py            # 路由到对应实例
  → application/ingress/handler.py           # 归一化 + 去重黑名单 + sibling-echo + emit
  → domain/lifecycle/events.emit_event       # 严格校验 kind → INSERT → _wake_or_inject
                                                ├── BLOCKED + 无 wake 进度 → 起线程 wake
                                                ├── RUNNING → mid-session signal_new_events
                                                └── wake in progress → return(等下个 emit/cron)
  → domain/lifecycle/scheduler.py            # 组装 wake prompt
  → infrastructure/ai/agent.py               # LLM 调用
  → interfaces/tools/*                       # 工具调用

兜底路径(emit→wake 因 race / 进程崩 / 自退避复活漏抓时):
  → infrastructure/scheduler/cron_lifecycle.py  # 60s tick pop_due_events
                                                 # → 共用 wakeup_policy.choose_reason → wake
```

核心改动总结(对比旧图):
- handler 不再看 affair 状态、不再起 wake 线程、不再做 urgency 二级分流
- emit_event 函数内部新增 `_wake_or_inject`
- 延迟完全归 `group_buffer.py`(消息系统),与事件层解耦
- 60s cron tick 从"主驱动"降为"兜底",作用是防止 emit→wake 漏抓

#### 主动休息链路

```
模型调 rest(until="...")
  → interfaces/tools/action_tools.py       # 解析参数 / 重复检测
  → domain/lifecycle/alarms.py             # 设置 timer 闹钟
  → domain/lifecycle/affairs/runtime.py    # 标 BLOCKED + set_wait_intent
  → agent session 结束
  → cron 到点 fire_at 触发 → 重新唤醒
```

### A.5 关键边界（不该跨的线）

- **`domain/` 只规划，不执行**：状态机、事件规则、记忆策略——但实际 LLM 调用、SQLite 写入、HTTP 发送都是 infrastructure 的事。
- **`application/` 不吞领域语义**：它编排，但不重新定义数字生命规则；比如 ingress 的"紧急度分类"是产品规则（属于 application 邻近 domain），但 rest/set_alarm 的语义定义在 domain。
- **`interfaces/` 不做决策**：飞书 adapter 只做协议转换 + 归一化；归一化后的语义判定（路由到哪个实例 / 是不是 sibling bot 自发）走 application。
- **`infrastructure/` 不定义数字生命语义**：它的 LLM client / SQLite driver / cron tick 都是通用工具；"何时唤醒""滑入什么 prompt""扣多少精力"这种规则在 domain。
- **跨实例隔离通过 ContextVar**：`set_current_instance_id` 在每个进程入口设置后，所有 db 连接、事件队列、token tracker 都自动按实例分流。开发者**不需要**手动传 instance_id 参数。

## 附录 B：事件概念分类卡

> 源码里有 4 个名字都叫 "event"——容易混淆。这一章用**「我是什么、我归谁管、我服务什么场景」**讲清楚。

### B.1 四个 "Event" 概念对照

| 名字 | 是什么 | 归哪一层 | 服务什么场景 |
| --- | --- | --- | --- |
| **`MessageEvent`** | 入口消息的归一化封装 | `application/ingress_interactions/` | 飞书 / CLI / Webhook / Scheduler 等任何入口的入站消息先归一化成它，再走标准消息工作流。 |
| **`EventInstance`** | 可注册和触发的生命周期工作项 | 类型注册在 `application/events/`；唤醒队列在 `domain/lifecycle/` | 第五章 "事件机制" 讲的就是它——数字生命的"心跳"。`group_message` / `timer` 等高峰值/N 分钟周期都是 EventInstance。 |
| **`FlowEvent`** | 跨层流转轨迹事件 | `domain/flow_event_log/` | 描述一次 wake run 内**发生了什么**——覆盖入口、记忆、编排、执行、反馈各层，用于事后审计和前端展示。 |
| **`ExecutionEvent`** | 执行层轨迹事件的一类 | `domain/execution/semantics/` | LLM 执行引擎内的细粒度事件——message trace / action / observation / agent error / rejection / state update / condensation 等。 |

读法：「**MessageEvent 是入口货、EventInstance 是生命脉动、FlowEvent 是事后审计带、ExecutionEvent 是 LLM 引擎内部黑盒**」。

### B.2 FlowEventLog

`FlowEventLog` 是**按 run 排序的跨层轨迹集合**。它在 `apps/{id}/data/state.db` 里独立表，**跟生命周期 `events` 表不冲突也不重叠**。

**必含字段**：`id` / `run_id` / `sequence` / `type` / `layer` / `source` / `timestamp` / `payload` / `summary` / `severity`

**引用字段**：`employee_id` / `message_event_id` / `parent_event_id` / `causation_event_id` / `correlation_id`

`layer` 限定 5 种：`ingress` / `memory` / `orchestration` / `execution` / `feedback`。

**Traceability 规则**：Action-like event 必须先记意图，再记 observation 或 error；Observation 通过 `causation_event_id` 或 payload 中的 execution / tool id 追溯到对应 action。

### B.3 各层不要混淆的边界

- `apps/{id}/data/state.db` 中的 **`events` 表**是数字生命的**生命周期触发队列**（第五章讲的队列）。
- `FlowEventLog` 在**同一 db**，独立表，**只用于审计**，不替换 `events` 表做唤醒判定。
- 具体执行引擎（`infrastructure/ai/agent.py`）不必定义 canonical event shape，但**必须遵守 FlowEvent + ExecutionEvent 契约**，这样未来替换执行引擎时不破坏审计能力。

### B.4 前端会话展示

员工控制台**会话 Tab** 的数据源是 `MessageEventDisplayView`（read model）——不是直接解析整段 raw prompt。规则：

- **User 起源的 visible message** 在一侧（人类发的话）
- **Employee/Assistant 的 visible message** 在另一侧（实例回的话）
- **`system_context` / `metadata` / memory / tool call-result / raw fallback** 作为独立 debug/context block 单独标灰
- **Markdown 必须来自 typed `MessageDisplayBlock`**——前端不做临时 raw prompt 解析

### B.5 稳定只读 API

- `GET /api/employee/{employee_id}/event-log/runs/{run_id}` → 按 sequence 排列的 FlowEventLog
- `GET /api/employee/{employee_id}/sessions/{session_id}` → 含 MessageEventDisplayView 的会话详情

字段契约由当前领域模型 + 应用读模型 + HTTP 路由共同定义；Spec Kit 产物**不**作为长期接口契约的唯一来源。

## 附录 C：自我迭代体系

> 实例可以自己注册新工具、新 skill、改 persona。这套「自我演化」体系是数字生命区别于普通 chatbot 的核心能力之一。
>
> 原则：**渐进披露**——不是把所有上下文都喂给模型，而是分层注入，让模型自己按需拉取。

### C.1 三层空间 + 两层注册

实例要"自己写代码自我进化"时，**它和系统共享一套三层空间结构**：

| 层级 | 目录 | 谁写 / 谁用 |
| --- | --- | --- |
| **个人** | `apps/<id>/tools/`、`apps/<id>/skills/` | 实例自己写、实例自己用（私有） |
| **项目** | `projects/<pid>/code/tools/`、`projects/<pid>/skills/` | 项目成员共享，用 `register_tool` / `register_skill` 注册到 manifest |
| **系统共享** | `shared/tools/`、`shared/skills/` | 跨项目，全实例可选启用 |

「**这条逻辑只有我用 → 个人空间；项目需要 → 项目空间；通用能力 → 共享空间**」——是实例每次动手前要回答的"放哪"问题。

### C.2 渐进披露的三层（Tier）

上下文注入按"必要性 + 体积"分 3 层：

#### Tier 0 — 每个 wake 必注（身份底色，~3KB）

| 段 | 是什么 |
| --- | --- |
| lifecycle | 系统怎么运转（行为规则、工具提示） |
| persona | 我是谁（名字、性格、工作习惯） |
| workspace | 我的工作空间在哪（简洁版，2 行） |

#### Tier 1 — 每个 wake 精简注入（摘要级，~2KB）

| 段 | 应该是什么形式 |
| --- | --- |
| 职责定位 | 1 行 / 项目（我担任什么） |
| 项目方向 | 1 句目标 + KPI 快照（不堆论断细节） |
| system_context | 最近 2 条 50 字内 lessons |
| session_digest | 1 句话最近做了啥 |
| consciousness（思绪残留） | 截 200 字 |
| chat_stream | 5 条 × 100 字 |
| task_board / my_context | 完整（模型 action 驱动） |

#### Tier 2 — 模型按需调（不占每 wake 配额）

| 想要什么 | 调什么 |
| --- | --- |
| 完整 thesis / KPI / 论断详情 | `sense_project_detail(project_id)` |
| 完整 chat 对话流水 | `sense_conversation(chat_id)` |
| 某实体的 profile + 记忆碎片 | `recall_entity(name)` |
| 某 skill 完整方法论 | `skill_view(name)` |
| 自己注册的全部工具清单 | `sense_my_tools` |

这套渐进披露是**为了对抗 prompt 膨胀**——上下文越长注意力越散，把核心放 Tier 0/1、细节放 Tier 2 让模型主动 fetch。

### C.3 Entity Recall 改造（breadcrumb 模式）

历史上每次 LLM call 前系统自动扫上下文 → 注入 3-5 条记忆碎片（大量 token）。改成 **breadcrumb 模式**：

- **自动注入只列名**：一行清单 `[联想命中] 华能蒙电 / 回测框架 / 龙头断板 — 需要详情调 recall_entity('name')`
- **模型主动按需 fetch**：调 `recall_entity('name')` 才返回实体的 profile + 关联碎片
- **模型决定相关性**：让模型判断"这个联想和我当前决策相关吗" → 调或不调

### C.4 manifest.json — 注册元数据

每个 scope（personal/project/shared）下维护一个 `manifest.json`：

```json
{
  "tools": [
    {"name": "ocr_extract_text", "version": "1", "description": "...", "registered_at": "...", "emoji": "📷"}
  ],
  "skills": [
    {"name": "leader_loop_strategy", "version": "1", "description": "...", "registered_at": "..."}
  ]
}
```

`interfaces/skills/__init__.py` 渲染 skill_index 时读 3 个 manifest → merge → 输出。模型自己注册过的 skill 自动出现在面板上——**不用改 prompt 文本**。

### C.5 模块边界

| 模块 | 职责 |
| --- | --- |
| `domain/capability/paths.py` | scope → 目录映射（纯函数，不碰 IO） |
| `domain/capability/store.py` | manifest.json 读写（metadata only） |
| `domain/capability/lifecycle.py` | `register_tool` / `register_skill` 全流程（检查 → 写文件 → 写 manifest → 热加载 handler code → 安全审计） |
| `interfaces/tools/capability_tools.py` | 薄薄一层 LLM 工具调度：校验 args → 调 domain lifecycle → 返 JSON |
| `interfaces/skills/__init__.py` | skill index 渲染（读 manifest + 系统目录 merge） |

**关键解耦**：工具系统（register → dispatch → 热加载 handler）和记忆系统（注册过的 skill 内容被 skill_view 拉取）**不互相调**，只通过 manifest 共享数据。

## 附录 D：实例数据布局

> 一个数字生命实例运行时会产生很多状态。这些状态按"节奏"分 db——同一节奏的数据放一起、共享一个 SQLite 文件。这一章讲每一类数据**为什么**这样放，schema 细节见代码（`infrastructure/persistence/instance/*.py`）。

### D.1 按节奏分库

每个实例目录（`apps/{id}/data/`）下分 **4 个 db 文件 + 1 个 markdown**：

| 文件 | 装什么 | 节奏 | 谁写 |
| --- | --- | --- | --- |
| **`state.db`** | 事件队列（events）、闹钟（timers）、affair 状态（affairs）、流转审计（flow_events） | 快、读写频繁 | cron + handler |
| **`messages.db`** | 该实例看到的所有 in/out 消息（text + chat + sender） | 中、append 为主 | ingress + express_to_human |
| **`sessions.db`** | LLM session 历史（prompt / response / tool_calls） | 中、append | agent.py（每个 LLM call 一次 row） |
| **`runtime_log.db`** | 唤醒审计：wake + turn + injection 三张表 | 快、append-only | scheduler.py（每次 wake） |
| **`persona/LIFE_PERSONA.md`** | 该实例的人格文本 | 慢、人改 | 用户 |

**为什么不一个 db 通吃**：SQLite 单写并发保护意味着一个表 lock 期间所有表都 lock。分库后，messages 大量写入时不会卡住 events 队列消费。**每个 db 的节奏越接近内聚越好**。

### D.2 跨实例消息总线（去中心化）

历史上有过"中央 conversations.db"设计——所有实例的消息都进同一个库。被去中心化消息总线替代：

- 每个实例独立 `messages.db`
- 实例 A 发消息到群 G → 写 A 自己的 `messages.db`（out）→ HTTP 广播给也在群 G 的实例 B → B 进程接收后写 B 的 `messages.db`（in）
- **没有共享库**，没有共享锁，多实例可独立并发执行

详细见 `docs/architecture/decentralized-message-bus.md`（后续会合并到主文档实体消息章节）。

### D.3 runtime_log.db 的 3 张表与 dedup

runtime_log.db 是审计 db，**只读为主**（前端 Sessions Tab 来源）：

#### `wake` 表

```sql
wake(id, instance_id, wake_seq, meta_json, started_at, ended_at)
```

`meta_json` 里塞各种 trigger / model / token / reason 字段（**不铺字段**就是为了加字段不用迁移 schema）。

#### `turn` 表

每次 LLM call 的一轮对话——`role` / `content` / `tool_calls` / `reasoning` / `token_count` / `error` 等。同一个 wake 可能有多 turn（多轮工具调用）。

#### `injection` 表

每次 wake 拼装 prompt 时注入的 system context 段——`sys_tool`（system_context / session_digest / consciousness / my_context 等）+ `content` + `injected_before_call`（计数 dedup 用）。

**dedup 策略**（按 sys_tool 名内置）：
- `latest`（删旧留新）：system_context / session_digest / consciousness / social_context / task_skill / my_context / task_board / wake_signal
- `append`（不去重）：chat_stream / entity_recall / narrative_* / 事件类注入

dedup **仅 wake 内**——上次 wake 的注入留作历史，下次 wake 在新位置重新加载。

### D.4 代码层 Ports

为了让上层（scheduler / agent）不直接绑死 SQLite 实现，定义了 4 个 Port Protocol：

```python
domain/persistence/
├── audit_port.py        # AuditWriter（wake / turn / injection 写入）
├── memory_port.py       # MemoryReader / Writer
├── vitals_port.py       # VitalsPort（精力读写）
└── workflow_port.py     # WorkflowPort（events / timers）
```

infrastructure 提供具体实现（`InstanceDB` 基类 + 4 个子 class）。**scheduler / agent 只 import port**，便于 mock + 单测。

### D.5 迁移历史提醒

历史上 contacts 曾单独 1 个 db（v1 设计），后来合入 `memory.db`——读写节奏跟 memory 接近，多 1 db 只多 1 connection 不带来收益。现在 contacts 作为 memory 的一部分（人脉 = 记忆的一种慢变量）。

迁移前的旧文件名（`session_db.py` / `hermes.db` / `tasks.db` / `contacts_v1_legacy` 等）属于 replacer 时代，**当前源码以本附录为准**。

## 附录 E：LLM 输入组装契约

> 你在控制台 Sessions Tab 点开一次 wake、查看"模型看到的完整输入"——这跟当时 LLM 真正收到的 messages 必须**一模一样**。否则所有"模型为什么会这样想 / 为什么没记住 X" 的判断都会错位。
>
> 历史上"发消息"和"前端回放"用两套组装逻辑，产生系统偏离：think 注入不一致、上下文压缩前端不还原、entity_recall 分别维护等等。**根治办法不是补全前端路径，而是让组装逻辑只有一份**——发消息和回放两路调用同一个函数。

### E.1 设计原则

1. **数据库是唯一事实来源**。组装永远只读库，不读运行期内存态、不接收运行期快照。
2. **组装是纯读函数**。同样的库内容 + 同样的 `(wake_id, call_seq)` → 同样的输出，可重放。
3. **所有运行期数据在组装前先落库**。事件消费、reasoning、segment narrative 都在调组装之前完成写库。
4. **写入（副作用）与组装分离**。写库只在发消息路径；组装不改库。

### E.2 关键边界参数：`llm_call_seq`

`llm_call_seq` 是一次 wake 内某次 LLM 调用的唯一稳定编号。所有分支都按它判边界，**绝不用 timestamp**——因为同一次 call 内多条 turn 时间戳极接近（同秒写入），无法区分 K 的输出 vs K+1 的输入。

**核心规则**：
- 调用 K 的**输入** = `llm_call_seq < K` 的所有 turn（前序各次 call 的完整往返）
- 调用 K 的**输出**落在 `llm_call_seq == K`，**不属于 K 的输入**
- K≥1 时 user 角色的 turn（防瞎忙警告）属于该次 call 的输入侧，**永不污染下游**

组装函数签名：

```python
assemble_llm_input(*, wake_id: int, call_seq: int, instance_id: str) -> list[dict]
```

### E.3 四类输入来源

**a. base turns（基础对话流）**

```sql
SELECT llm_call_seq, position_in_call, role, content, tool_name, tool_call_id, tool_calls, reasoning
FROM turn
WHERE wake_id = ? AND llm_call_seq < ?
ORDER BY llm_call_seq, position_in_call, id
```

**b. injections（system_context / session_digest / consciousness / chat_stream 等）**

```sql
SELECT injected_before_call, sys_tool, scope_id, content
FROM injection
WHERE wake_id = ? AND injected_before_call <= ?
ORDER BY id
```

**dedup 在写库阶段完成，组装不做二次 dedup**——表里保存的就是 dedup 后的最终状态，按 `ORDER BY id` 行序重放即可。

**c. reasoning（think 历史）**

```sql
SELECT reasoning FROM turn
WHERE wake_id = ? AND llm_call_seq < ? AND role = 'assistant'
  AND reasoning IS NOT NULL AND reasoning != ''
ORDER BY llm_call_seq ASC, position_in_call ASC, id ASC
```

取末尾 12 条，传给 `provider.inject_into_messages(max_rounds=10)`。

**d. segment narratives（上下文压缩叙事）**

只在该 wake 该 call 的 total token 估算超过压缩阈值时参与：

```sql
SELECT llm_summary FROM memory_layers
WHERE layer = 'segment' AND period = '{session_id}#{segment_index}'
```

段切分按 user 消息为段起点 marker，段的 session_id / segment_index 从段内 turns 推断。回溯对一段当时是否被压缩的判断 = 该段 narrative **是否存在于库**——存在视为已压缩，不存在视为原始 turns。

### E.4 组装顺序（绕不开的代码级公式）

```
final = []
# 1. system（来自 wake.meta_json.system_prompt_text）
# 2. continuation_history（来自 wake.meta_json.continuation_history）→ 转 user/assistant/tool
# 3. 交错拼装 base turns（不含 think）:
#    a) apply injections WHERE injected_before_call == 0
#    b) apply call 0 的 user action prompt（WHERE llm_call_seq==0 AND role=='user'）
#    c) for k in range(1, call_seq + 1):           # 含 call_seq
#          apply call (k-1) 的 assistant+tool turns（WHERE llm_call_seq==k-1
#          AND role != 'user'）  # 这是 (k-1) 的输出，是 k 的输入
#          apply injections WHERE injected_before_call == k
# 4. provider.inject_into_messages(final, reasoning, max_rounds=10)   # think 拼回
# 5. final = compact_if_needed(final)                                  # 压缩叙事
return final
```

**5 个关键点**：

1. 循环 `range(1, call_seq+1)` **含 call_seq** —— 当 `k==call_seq` 时 apply 的是 `call_seq - 1` 的 assistant+tool turns，是本次 call 的最后一次"前序输出"。`call_seq` 自身的 turns 永不被施加。
2. **call 0 的 user action prompt 显式 apply**：仅 `role == 'user'`；call 0 的 assistant/tool 是它自己的输出，不施加。
3. **中间 call 的 user 跳过**：`role != 'user'` 过滤（防瞎忙警告仅归属自己的 call 输入侧）。
4. **think 注入必须在 injections 之后（步骤 4 后于步骤 3）**：原因——fake-assistant 行 `content=None` 且带 `tool_calls`，若 think 先注入会误把文本贴到 fake-assistant 前。同一原因 `provider.inject_into_messages` 必须加守卫：**跳过 `_is_fake=True` 的 assistant 行**（这一守卫之前缺失）。
5. **压缩放在 think 注入之后**，与运行期 `_maybe_compress_messages` 在 `_chat` 最后执行一致。

### E.5 组件归属

| 模块 | 职责 |
| --- | --- |
| `infrastructure/ai/assembly.py` | **唯组合函数** `assemble_llm_input()`，纯读库 |
| `infrastructure/ai/providers.py` | think 注入子步骤 `inject_into_messages`，被组装函数调用 |
| `infrastructure/ai/agent.py` | 发消息路径编排——副作阶段（落库）→ 组装阶段 → HTTP 发送 |
| `infrastructure/persistence/instance/runtime_log.py` | `render_input_for_call()` 改为薄包装 |
| `interfaces/web/employee-console/` | 只渲染 `assemble_llm_input()` 字面输出，不重组装 |

### E.6 确定性约束

为了保证发消息和回溯**字面一致**：

1. **fake tool_call id 用库 row id 做种子**：`fake_{sys_tool}_{injection.id}`、`narrative_{segment_index:03d}`。不依赖运行期计数器。
2. **JSON 序列化稳定**：`tool_calls` 入库时 `json.dumps(ensure_ascii=False, default=str)`，回溯 `json.loads` 一致。
3. **dedup 只在写库阶段做**：`runtime_log.DEDUP_STRATEGY` 是写库规则（`inject()` 内置 DELETE+INSERT），assembly 不读这个常量、不做二次 dedup。
4. **不依赖 timestamp 边界**：`turn.timestamp` 仅用于排序同 `llm_call_seq` 内多条，不用于"调用边界"。
5. **provider.inject_into_messages 必须加守卫**：跳过带 `_is_fake=True` 的 assistant。配套：`_injection_as_fake_pair` 给 assistant_msg 也打上 `_is_fake=True`。

### E.7 回溯正确性自校

部署后用真实 wake 做字面对照验证：
1. 选一个调用档保留完整的 wake（`turn / injection / memory_layers` 三表行齐全）。
2. `assemble_llm_input(wake_id=W, call_seq=k)` 跑回溯。
3. 跟发消息时实际发的（阶段 1 后 dump 一份比对）逐字 diff。
4. 预期完全一致；唯一历史差异源 `_sys_tool_counter → segment_index` 已修复。

完成自校后，前端"完整详情"才能被信任为字面回放，调试才有效。

## 附录 F：记忆代谢体系

> 数字生命一跑起来就会不断产生记忆碎片：今天的教训、近期的顿悟、晚上的复盘日记、临时记下的"X 股明天盯一下"等。如果只有写入没有代谢，两个月后记忆文件会涨到几万字，里面**一半是重复**、一半过时——模型每次 wake 的 prompt 就被这些垃圾挤压，注意力涣散。
>
> 这一章讲怎么让记忆**自己活/自己死**——三条轴：类型轴决定"怎么写怎么管"，实体轴决定"怎么找怎么联"，代谢流决定"什么时机清什么"。

### F.1 三个根因

旧系统只有"写入"没有"代谢"：

| 问题 | 表现 |
| --- | --- |
| 只有写入没有代谢 | LESSONS 30 条中 15+ 条重复；RULES 12 条中同一规则被写 5 次 |
| 没有晋升和治理管道 | 每条记忆只有出生机制，无碎片→经验→规则的晋升 |
| 没有实体索引 | 按文件名分类（RULES/LESSONS…），但无法按"华能蒙电"这种主题索引 |

### F.2 双轴架构

两条轴正交——一条记忆同时有"类型"和"实体"：

```
类型轴（决定怎么写/怎么管）             实体轴（决定怎么找/怎么联）
─────────────────────────             ─────────────────────────
RULES    → 全量注入，周度审计          一条记忆关联一组实体
LESSONS  → 最近 3 条注入，周度晋升      实体 → 倒排索引 → 快速匹配
CONTEXT  → 全量注入，每次覆盖          不同记忆通过实体互相链接
CONSCIOUSNESS → 不注入，按需读取
SCRATCHPAD → tail 500，三重出口代谢
```

### F.3 实体索引

存储位置：`apps/{id}/data/memories/entity_index.json`。

每条记忆可以挂一组 `entities=["华能蒙电", "600863", "涨停策略"]` 等。系统建倒排索引把实体名 → 关联记忆 list 反向挂。

**写入时机**：模型调 `add_lesson` / `record_thought` / `update_rules` / `write_diary` 时可选传 `entities`。prompt 鼓励但不强制。

**多维排序公式**（如何选 top N 记忆召回）：

```
score = recency × 0.35 + authority × 0.25 + context_overlap × 0.25 + verification_bonus × 0.15
```

| 维度 | 含义 | 范围 |
| --- | --- | --- |
| recency | 时间新鲜度，30 天线性衰减 | 0.0–1.0 |
| authority | 类型权威度：RULES=1.0 / LESSONS=0.8 / CONSC=0.6 或 0.3 | 0.2–1.0 |
| context_overlap | 当前上下文实体与记忆 linked_entities 的 Jaccard 相似度 | 0.0–1.0 |
| verification_bonus | 被确认次数 / 5，上限 1.0 | 0.0–1.0 |

**三层防过时**：
1. 时间衰减（recency 权重 0.35）
2. 类型区分（trading_wait 类 consciousness authority 只有 0.3）
3. 显式废止（通过 `supersedes` 参数 ——未来特性）

### F.4 双时态召回

**Wake 时批次召回**：scheduler 构建 prompt 时 → strings 匹配已知实体 → `query_entities_ranked(limit=5)` → 注入 `[实体触发记忆]` 块。
**Mid-session 快速联想**：AIAgent 每次 LLM 调用前 <5ms 扫描，注入一行 `[快速联想] 华能蒙电 / 回测框架`，model 自己判断要不要调 `recall_entity('name')` 拉详情。

去重粒度：entity 级（同 session 同实体只注入一次）+ memory 级（同 memory_id 只注入一次）。

### F.5 四条代谢流

| 时机 | 做什么 | 触发点 |
| --- | --- | --- |
| **写入时去重** | `record_thought` Jaccard > 0.7 跳过；`add_lesson` 与最近 5 条相似度 > 60% 合并（追加"再次确认"）；`update_rules` 检 `## 场景：XXX` 标题重复 | 模型调工具瞬间 |
| **Scratchpad 三重出口** | ① 每次 session 自审 prompt 鼓励搬走；② 晚间复盘末尾归类；③ 兜底代码 `>5000 字符自动截断到 2000` | 多个触发点 |
| **日记整合**（晚间自动） | `write_diary()` 在 21:00 后自动调 `consolidate_day_diary()`，整合碎片日记 + 意识残留 + session 摘要 + 实体热力图 | evening_review routine |
| **Lesson → Rule 晋升**（周度） | weekly_review Phase 2.5 合并相似；Phase 3 晋升标准：被验证 2+ 次 + "每次应遵守的" + 违反有损失。降级：连续两周未触发 / 每次违反但无后果 / 措辞模糊 | weekly_review routine |

读法：「**写入瞬间就拦重、晚间合并碎片、周末沉淀规则**」——三层节奏对应人脑"顿悟 → 笔记 → 反思 → 沉淀"的完整链路。

### F.6 自我认知（SELF_KNOWLEDGE）

除了 RULES / LESSONS，还有一类 self-knowledge：

- **不是规则**（不规定行为），**不是经验**（不总结外部教训）
- 是**对自己行为模式的中立观察**：「在 X 情况下，我倾向于 Y」
- 写入：weekly_review 时由模型判断追加
- 注入：self_iteration task / initiative 唤醒时按需注入，不全量
- 存储位置：`apps/{id}/data/memories/SELF_KNOWLEDGE.md`

### F.7 记忆文件三层架构

**第一层：行为约束（每次 wake 注入）**

| 文件 | 注入 | 治理 |
| --- | --- | --- |
| `RULES.md` | 全量 | 周度审计 + 写入去重 |
| `LESSONS.md` | 最近 3 条 | 周度合并 + 晋升 + 过期 |
| `CONTEXT.md` | 全量 | 每次覆盖 |

**第二层：叙事/检索（按需注入）**

| 文件 | 注入 | 治理 |
| --- | --- | --- |
| `CONSCIOUSNESS.md` | 上次残留或截 200 字 | 每次 session 末尾写 |
| `SELF_KNOWLEDGE.md` | self_iteration / initiative 才注入 | weekly_review 追加 |
| `SCRATCHPAD.md` | tail 注入 | 三重出口（章节 5.2） |
| `diary/YYYY-MM-DD.md` | 不注入，按需 `recall_dated_diary` | 晚间自动整合 |

**第三层：历史归档（不再注入）**

| 文件 | 治理 |
| --- | --- |
| `diary/archived/` | 周日归档当周日记 |
| `memories/_archive/` | 周度 lessons 降级（写"过期"标签后挪入） |

读法：「**活着的不超过一周，叙事的不超过 24 小时，历史的不进 prompt**」。

## 附录 G：多 Agent 协作设计

> 一个数字生命实例孤立运行是一回事，多个实例（zero / alpha / beta…）协作跑同一个项目是另一回事。这一章讲怎么让"数字员工"形成真正的团队——而不是各干各的。

### G.1 参与者与组织单位

**参与者（actor）** 有 3 种：

- **实例（数字员工）**：跑着的 LLM agent 实例，有 persona / 状态 / 工具调用能力。
- **真人**：你（开发者 / 产品负责人），通过飞书等通道用消息参与。
- **系统**：cron / scheduler / ingress / budget gate 等"员工看不到但确实存在"的后台主体。

**组织单位**有 3 个层级：

- **实例个人**：一人格一份运行数据，独立的 persona / memory / 状态。
- **项目（project）**：多 actor 协作的最小单元。一个项目定义一组参与者、岗位、成果目标和论断假设。
- **项目内岗位**：实例在项目里担任的具体角色（策略师 / 架构师 / 执行者 / 交易员等）。岗位规定**这个实例该把什么放在心上**。

### G.2 项目怎么启动

```
真人 / 规划者发起 create_project
  ↓
系统自动创建根任务（项目概述 + 目标作为 title/description）
  ↓
按预置模板创建初始子任务：
  - "项目分工"（承担 = 项目经理 instance_id）
  - "项目管理"（承担待分配）
  ↓
项目经理自己 wake 后主动拆「项目分工」todo，分配给其它实例
  ↓
被分配到的实例下一个 wake 看到自己名下多了一条 todo
```

「**项目不是配置出来的，是被规划者拆出来的**」——这是设计核心。

### G.3 协作型待办（待办的双向性）

待办不只是"我手头的事"。**协作型待办**有两种走法：

- **分配给别人的待办**：creator=我，assignee=alpha。我创建后 alpha 下次 wake 就看到。我可以通过 `todo list(assignee="alpha")` 跟进进度。
- **共同待办**：creator=我 + assignee=我（持续型，每周 review 由自己推进）。

读法：「**待办是独立 entity，关联关系是属性——不是归属**」。todo 可以跨实例流转，creator / assignee / project 是它的 metadata。

### G.4 系统怎么转起来（6 大推动力）

数字生命系统不是"被动等指令"——而是被 6 种力量推动前进。按"自定义程度从高到低"：

| 推动力 | 是什么 | 强度 |
| --- | --- | --- |
| **任务自驱（最核心）** | 承担者的当前 todo 到点 / 条件成立 → wake → 推进 | 最高 |
| **项目层节奏** | daily sync / weekly review / monthly milestone——根任务的触发器 | 高 |
| **个人层节奏** | morning_plan / afternoon_checkin / evening_review 三个 routine | 中 |
| **业务事件** | 真人消息 / sibling 实例协作 send | 高 |
| **真人指令** | 真人主动 set_task / 干预 | 最高 |
| **主动探索** | 精力充沛且空闲时自驱"扫看板挑该做的" | 低（背景层） |

**关键约束**：「员工不自由发挥'我该做什么'」。每个实例都有明确的岗位 + 待办 + 项目归属。它在 wake 时根据**面板上的信号**（最高优先级过期 todo / message / routine）选当前动作——不会"觉得"自己该找个新事做。

### G.5 会议（自然形成的协作）

不刻意搞"会议 module"，让会议从协作机制中自然涌现：

**日会/周会怎么自然形成**：
- daily routine（早间规划）→ 项目经理发起一段"同步今日进度"，所有成员各自表态。
- weekly review（周日）→ 项目经理 check thesis 是否成立 → 公布决议。

**会议通过消息进行**：
- 实例 express_to_human 到群 @ 其它成员 → 群消息事件唤醒他人 → 他人回应。
- 不需要专门的"会议 API"——消息通道就是会议通道。

**Skill 引导会议结构**：
- `project_management / self_review / entity_curation` 等 skill 提供方法论。
- 模型按需 `skill_view("xxx")` 拉取，按其中模板推进。

**会议产出新任务新待办**：
- 实例讨论中想到的事 → 立刻 `todo create` 落地 → 任务树更新。
- 讨论的决议 → 写入项目 `decisions.md`（项目 docs）。

### G.6 反思与沉淀（自然 vs 强制）

#### 任务完成时的反思（自然发生）

任务完成后工具返还数据带 prompt 提示"沉淀"——但不强制：

- 模型决定要不要 `add_lesson`
- 系统根据 `verification_count` 自动权重提升（被多次验证同条 lesson → 周度晋升为 RULE）
- 没沉淀 → 0 损失；沉淀了 → token 花得值

#### 自然反思 vs 强制反思

| 类型 | 触发 | 用什么 skill |
| --- | --- | --- |
| **自然反思** | 任务 done 时 prompt 隐含鼓励 | 内联 |
| **晚复盘（半强制）** | 21:00 evening_review routine | `self_review` skill |
| **周复审（强制）** | 周日 21:00 weekly_review routine | `self_review` + `entity_curation` |
| **每月里程碑** | 1 号 09:00 monthly milestone | 全套 + 论断 review |

#### 教训的反向影响

学到一条 `lesson` 后，它**自动**进入下次 wake 的 `system_context` 注入（最近 N 条），不需要任何额外配置。模型下次执行类似动作时就会"想起"——这是规则系统天然起作用的方式。

### G.7 实施路线（6 阶段 P0-P5）

不在本附录展开实施细节。整体阶段：

1. **P0**：修正"任务成果 → 状态联动"基础（done 时自动 hook）
2. **P1**：引入 todos 表
3. **P2**：任务类型 + Skill 绑定（task.type → 推荐 skill）
4. **P3**：项目根任务 + 初始任务树自动建（create_project 自动建"项目分工"等）
5. **P4**：scheduler wake 时拼上下文时把"当前激活 todos / 父任务链 / 项目 KPI / 推荐 Skill"一起注入
6. **P5**：pilot 试跑两个项目——「数字员工迭代」(内部 metaproject) + 「模拟炒股」

### G.8 不在范围（未来）

- 跨项目资源调度（同一实例同时承担多项目时的预算分摊）
- Metric 反馈闭环（教训 → 可量化度量 → 度量驱动改进）
- 真人介入工具（拍板按钮、暂停闸等增强真人权威）

## 附录 H：去中心化消息总线

> 一个群里同时挂多个数字员工（zero / alpha）时，他们之间怎么分工、谁看到那条消息、谁能响应、是否会重复刷屏？这一章讲清楚"为什么不搞中央消息库"。

### H.1 旧方案的维护债

历史上用过「统一聚合库」方案——所有实例共享 `var/conversations/chats.db`，配 `unified_contacts` 跨实例 ID 映射 + fan-out 事件分发。表面干净，实际维护成本不可接受：

| 旧方案组件 | 问题 |
| --- | --- |
| `chats.db` 共享库 | 多 writer 竞争（即便上 WAL 只是补丁） |
| `unified_contacts` 表 | 跨实例 ID 映射是**伪问题**——实例视角的 open_id 永不外输，名字冲突用 per-instance `contacts.notes` 解决即可 |
| `_fan_out_to_other_instances()` | fan-out 用 events 表 INSERT → 违背"receiving 实例应该自己决定 urg 是否 wake"的产品意图 → race condition |
| 共享 DB 物理路径 | 实例必须同机 → 不能跨机部署 |
| 新平台接入 | unified_contacts 内卷成 N 维矩阵（lark + wechat + dingtalk…） |

### H.2 新方案：每实例独立消息库 + 同群广播

**核心原则**：每个实例独立维护自己的消息库。**不共享任何 DB 文件、不强求全局统一 ID**。跨实例同步**只发生在同一个群内**，通过实例之间直接 HTTP 广播（无第三方 broker）。

```text
飞书群 G 同时挂 zero + alpha
  ↓
平台推一条消息
  ↓
zero bot 的 WS 收到 → 写 zero.messages.db（in）→ zero.urgency 判断
alpha bot 的 WS 也收到 → 写 alpha.messages.db（in）→ alpha.urgency 判断
  ↓
两边独立处理，互不通气
```

实例上飞书发的消息要广播给同群的兄弟实例：

```text
zero express_to_human(text="...", chat_id=群G)
  ↓
zero.messages.db 写一条 out
  ↓
zero 后台 HTTP POST → 群 G 里的 alpha 实例
  ↓
alpha 接收 → 写 alpha.messages.db（in, source=broadcast:<zero_iid>）
  ↓
alpha 走跟飞书入站完全一样的 urgency 分类
  ↓
@alpha 的消息 → immediate；否则 → 20-40s 累积
```

### H.3 与产品方对齐的 6 个决策

| # | 决策 | 含义 |
| --- | --- | --- |
| 1 | 广播走 urgency 判断（不特殊化） | zero 广播给 alpha，alpha 收到后跟飞书入站走**完全一样**的 urgency 分类。提到 alpha → 立即响应；否则进 20-40s 累积窗口 |
| 2 | 双 bot 同消息各自收 | zero 和 alpha 各收各的飞书推送，**入站路径完全不通信**。重复响应全靠 urgency 降级 + fuel token gate 防护 |
| 3 | 不处理跨平台同名 | 每实例自己库的 open_id 自洽不外漏。名字冲突靠 per-instance contacts.notes 区分（前端可编辑）让模型识别。**unified_contacts 整套删** |
| 4 | 偶发丢失可接受 | HTTP 广播 fire-and-forget，失败只 log。peer 离线时丢几秒上下文，下次自然补上。不引入复杂 ACK/重传 |

### H.4 数据模型

每实例独立 `apps/<id>/data/messages.db`，单表：

```sql
CREATE TABLE messages (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    ts              TEXT NOT NULL,
    direction       TEXT NOT NULL CHECK (direction IN ('in', 'out')),
    source          TEXT NOT NULL,                 -- lark / broadcast:<instance_uuid>
    chat_id         TEXT NOT NULL,                 -- 平台原始 chat_id
    platform_sender TEXT NOT NULL DEFAULT '',      -- per-app open_id，永不跨实例输出
    sender_name     TEXT NOT NULL DEFAULT '',      -- 人类可读名
    sender_role     TEXT NOT NULL DEFAULT '',      -- 'self' / 'human' / 'bot-broadcast' / 'other'
    text            TEXT NOT NULL,
    msg_ref         TEXT NOT NULL DEFAULT '',
    UNIQUE (source, msg_ref)                       -- 同一原始消息去重
);
```

**双库职责清晰**（不混淆）：

| 库 | 装什么 | 用途 |
| --- | --- | --- |
| `messages.db` | 对外消息流水（in/out 一份） | 前端 Sessions Tab + LLM `chat_stream` 注入 |
| `sessions.db.messages` | 模型上下文 turn 序列（system / user / assistant / tool 历史） | 给模型拼续接历史 |

两库的数据可能重叠（模型的话也是对外消息），但救不同场景——任务不同。

### H.5 入站 / 出站路径

#### 入站

handler 收到飞书消息：
1. INSERT `direction='in', source='lark', chat_id=oc_xxx, sender_name=飞书给的name`（`UNIQUE(source, msg_ref)` 保证同一飞书消息被多个实例各记一次也不冲突，因为每个实例独立库）
2. 触发该实例的 urgency 判断：@ 我 / 含 attention_keywords / 群 owner 说话 → immediate → 立即 wake；否则 → soft → 20-40s 累积
3. **不再 fan-out 给其他实例**——其他实例会由它们自己的 adapter 独立收到同一条平台消息

#### 出站

实例发言 → HTTP 广播给同群兄弟 → 接收方写 messages.db + 走完全相同的 urgency 分类。

### H.6 一致性边界

| 场景 | 是否处理 |
| --- | --- |
| 同一飞书消息在 zero / alpha 各记一条 | **正常**——两个独立 INSERT 各自入库 |
| 实例广播给同群但接收方离线 | fire-and-forget 丢了就丢了，下次 cron 自然补 |
| 跨平台同名识别（"张三" 是哪个张三） | **不处理**——用 per-instance `contacts.notes` 字段让模型区分 |
| 同 bot 实例发自己消息循环 | 双 bot 检测：如果 `sender_id == self.bot_id` 直接 return，不入 messages.db 也不 emit event |
| 多 bot 同步发言（race） | urgency 20-40s 累积窗口 + 累积去重，让 responding 实例有时间互看 → 减少重复 |

### H.7 与附录 D 实例数据布局的关系

各实例的 messages.db 已在附录 D 的 4 个 db 中描述（"中"节奏）。本附录补充**跨实例行为**——broadcast 协议、urgency 一致性、双 bot 防环等。
